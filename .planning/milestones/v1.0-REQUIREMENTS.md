# Requirements: Cloud CLI Proxy

**Defined:** 2026-03-25
**Core Value:** 给每个用户提供一台开箱即用的 SSH 云主机，并且严格保证其所有出网流量都走受控的指定出口 IP，同时保持“一条命令启动”的体验足够顺滑。

## v1 Requirements

### 启动接入

- [x] **ACCS-01**：用户可以执行一条简短的 `curl` 启动命令，并在终端中看到用户名密码输入提示。
- [x] **ACCS-02**：用户在凭证正确后，可以在终端中看到云主机启动中的状态提示。
- [x] **ACCS-03**：当云主机已就绪且用户有权限时，系统可以直接把用户接入一个可用的 SSH 会话，而无需手工查找主机和端口。
- [x] **ACCS-04**：当凭证错误、账号过期、未绑定出口 IP 或启动失败时，用户能收到明确、适合终端展示的错误提示。

### 用户与生命周期

- [x] **USER-01**：管理员可以创建、禁用和删除用户账号。
- [x] **USER-02**：管理员可以设置或轮换用户在启动流程中使用的登录密码。
- [x] **LIFE-01**：管理员可以启动某个用户的容器化云主机。
- [x] **LIFE-02**：管理员可以停止某个用户的容器化云主机。
- [x] **LIFE-03**：管理员可以基于受管镜像模板重建某个用户的云主机。
- [x] **LIFE-04**：管理员可以为用户账号或主机分配设置到期时间。
- [x] **LIFE-05**：已过期用户不能再开启新会话，并且后台能明确显示其已过期状态。

### 运行环境

- [x] **RUNT-01**：每个用户会话都由一个独立的 Docker 容器承载，且该容器来自受管镜像模板。
- [x] **RUNT-02**：受管镜像模板内预装 OpenSSH Server、基础 Shell 工具和 `claude code`。
- [x] **RUNT-03**：在把用户接入会话前，平台会验证容器和 SSH 服务都已经真正就绪。

### 出口网络控制

- [x] **NET-01**：每个可运行容器都必须至少绑定一个出口 IP 资源；如果未绑定，启动必须失败。
- [x] **NET-02**：用户容器中的所有出站流量都必须被强制导向指定的全隧道路由路径。
- [x] **NET-03**：用户容器中的 DNS 解析也必须走受控路径，不能回落到宿主机或 Docker 默认解析器。
- [x] **NET-04**：对用户容器而言，任何未走隧道的出站流量都必须被默认阻断。
- [x] **NET-05**：在主机被标记为可接入前，系统会验证出口 IP 和 DNS 路径都符合预期。

### 管理后台控制面

- [x] **ADMN-01**：管理员可以创建、编辑、启用、禁用和删除出口 IP 资源。
- [x] **ADMN-02**：管理员可以把一个或多个出口 IP 资源绑定到用户或用户主机策略上。
- [x] **ADMN-03**：管理员可以在管理系统中查看用户、容器、出口 IP 绑定、生命周期和到期状态。
- [x] **ADMN-04**：管理员操作和启动结果会被记录为运维事件，便于排障和支持。

## v2 Requirements

### 商业化与平台扩展

- **COMM-01**：运营方可以管理计费计划、余额和支付状态。
- **COMM-02**：用户可以在没有管理员介入的情况下完成自助账号操作。
- **PLAT-01**：平台可以把用户调度到多台宿主机或多个节点上。
- **PLAT-02**：平台在 SSH 之外还支持浏览器终端接入。
- **PLAT-03**：平台可以按客户类型提供多套镜像模板或工作区配置。

## Out of Scope

| 功能 | 原因 |
|------|------|
| 计费和套餐管理 | 在核心主机生命周期和网络强约束能力被验证前，先不引入商业化复杂度 |
| 多宿主机调度 | v1 明确限定为单宿主机，优先验证主链路 |
| 浏览器终端 | MVP 的目标访问模型是 SSH，而不是多入口并行 |
| 用户自定义任意镜像 | 这会削弱就绪性、安全性和可支持性 |
