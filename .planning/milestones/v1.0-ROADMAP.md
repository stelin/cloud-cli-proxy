# Roadmap: Cloud CLI Proxy

## 🚧 v1.0 MVP

**Milestone Goal:** 在单宿主机上交付可自用的容器化 SSH 云主机 MVP，先完成控制面、运行时、网络强约束、启动接入、后台管理与上线前硬化。

## Overview

通向 v1 的路径，是先把单宿主机上的容器化 SSH 云主机控制面和网络强约束做对，再去打磨运营和用户体验。我们会先建立运行时基础、权限边界和受管镜像，然后完成出口 IP 强制绑定与全隧道网络，再实现 `curl -> 登录 -> loading -> SSH` 的接入流程，随后交付后台管理、到期治理和审计能力，最后完成硬化与上线准备。

## Phase Details

### Phase 1: 基础控制面与主机代理
**Goal**: 建立单宿主机控制面的基础能力、受管镜像、持久化存储，以及宿主机特权操作边界。
**Depends on**: Nothing (first phase)
**Requirements**: RUNT-01, RUNT-02
**Success Criteria** (what must be TRUE):
  1. 运维方可以在单台宿主机上部署控制面和持久化数据库。
  2. 系统可以基于项目镜像模板创建受管用户容器。
  3. Docker 与网络等特权操作被隔离在明确的宿主机边界之后。
  4. 用户、主机、出口 IP 资源和任务等核心数据可以持久化保存。
**Plans**: 3 plans

Plans:
- [x] 01-01: 搭建控制面、数据库结构和基础服务布局
- [x] 01-02: 构建带有 OpenSSH 和 `claude code` 的受管用户镜像
- [x] 01-03: 引入主机代理和运行时特权边界

### Phase 2: 隧道出网强制层
**Goal**: 保证每个可运行用户容器都绑定出口 IP，并且只能通过受控隧道出网。
**Depends on**: Phase 1
**Requirements**: NET-01, NET-02, NET-03, NET-04, NET-05
**Success Criteria** (what must be TRUE):
  1. 没有出口 IP 绑定的容器会在启动阶段立即失败。
  2. 运行中的用户容器对外显示的出口 IP 与分配结果一致。
  3. 用户容器中的 DNS 解析也走受控路径，而不是宿主机或 Docker 默认解析器。
  4. 所有未经过隧道的直接出站流量都会被策略阻断。
**Plans**: 3 plans

Plans:
- [x] 02-01-PLAN.md — 数据层与绑定校验：网络类型/错误体系、DB 隧道字段扩展、启动前绑定校验和 --network=none
- [x] 02-02-PLAN.md — 隧道接线与默认拒绝：netns/WireGuard/veth 注入、nftables 默认拒绝、DNS 受控路径和 TunnelProvider
- [x] 02-03-PLAN.md — 三重校验与就绪判定：出口 IP/DNS/泄漏三重校验、事件记录集成和预检扩展

### Phase 3: 启动入口与 SSH 接入
**Goal**: 在安全运行时之上，构建顺滑的终端启动流程，完成认证、启动中提示和 SSH 接入。
**Depends on**: Phase 2
**Requirements**: ACCS-01, ACCS-02, ACCS-03, ACCS-04, RUNT-03
**Success Criteria** (what must be TRUE):
  1. 用户可以执行短 `curl` 入口并在终端中完成认证。
  2. 用户在主机准备过程中能看到明确的进度提示。
  3. 只有当主机真正准备完成后，用户才会被接入到可用的 SSH 会话。
  4. 凭证错误、账号过期、绑定缺失和启动失败都会返回清晰的终端提示。
**Plans**: 3 plans

Plans:
- [x] 03-01: 构建启动认证、会话任务和终端输出流程
- [x] 03-02: 实现启动进度展示和基于就绪门槛的接入控制
- [x] 03-03: 完成 SSH handoff、重试机制和失败体验

### Phase 4: 后台管理界面
**Goal**: 让运营方可以通过管理系统完成用户、密码、出口 IP 资源、绑定关系、生命周期操作和状态查看。
**Depends on**: Phase 3
**Requirements**: USER-01, USER-02, LIFE-01, LIFE-02, LIFE-03, ADMN-01, ADMN-02, ADMN-03
**UI hint**: yes
**Success Criteria** (what must be TRUE):
  1. 管理员可以创建、禁用和删除用户。
  2. 管理员可以轮换用户的启动凭证。
  3. 管理员可以创建出口 IP 资源并将其绑定到用户或主机策略。
  4. 管理员可以启动、停止、重建用户主机，并查看当前状态。
**Plans**: 3 plans

Plans:
- [x] 04-01-PLAN.md — 搭建后台认证外壳、基础布局和状态看板（JWT 认证 + React SPA 脚手架 + 仪表板概览）
- [x] 04-02-PLAN.md — 实现用户与凭证管理（用户 CRUD API + 密码轮换 + 用户管理页面）
- [x] 04-03-PLAN.md — 实现出口 IP 绑定与主机生命周期操作（出口 IP CRUD + 绑定管理 + 主机启停重建 + 任务列表）

### Phase 5: 到期、审计与清理
**Goal**: 实现到期治理，并补齐日常运营所需的事件记录和运行状态清理能力。
**Depends on**: Phase 4
**Requirements**: LIFE-04, LIFE-05, ADMN-04
**UI hint**: yes
**Success Criteria** (what must be TRUE):
  1. 已过期用户无法再发起新的启动会话。
  2. 属于已过期用户的运行中主机会按照策略被处理。
  3. 认证、启动、生命周期和到期事件都会被记录并可查看。
  4. 陈旧任务和运行时漂移状态可以在不手工 SSH 排障的前提下完成对账。
**Plans**: 3 plans

Plans:
- [x] 05-01-PLAN.md — 数据层、调度框架与到期执行（DB 迁移 + scheduler 包 + expiry 扫描器 + admin API 到期支持）
- [x] 05-02-PLAN.md — 事件记录注入、事件查询 API 与运行时对账（admin 事件注入 + events API + reconciler + agentapi inspect）
- [x] 05-03-PLAN.md — 前端到期展示与事件日志（用户 expiry UI + 事件日志页面 + 仪表板事件摘要）

### Phase 6: 加固与 MVP 就绪
**Goal**: 让产品达到可以稳定自用并交给第一批真实客户试用的程度。
**Depends on**: Phase 5
**Requirements**: ACCS-01, ACCS-03, NET-05, ADMN-03, ADMN-04

  3. 单宿主机部署流程和运维操作手册已经成文。
  4. MVP 可以交给真实用户使用，而不依赖隐藏的人工步骤。
