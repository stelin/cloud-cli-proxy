## 出口 IP 绑定语义

| Option | Description | Selected |
|--------|-------------|----------|
| 单主机单活跃出口 IP | v1 运行时只使用一个已绑定 IP，切换走明确动作 | ✓ |
| 多绑定，但始终选主绑定 | 允许预绑多个 IP，但正常运行只认主绑定 | |
| 多绑定并自动切换 | 可在多个已绑定 IP 之间自动切换 | |

**User's choice:** 单主机单活跃出口 IP
**Notes:** 运行时只认一个生效出口，不做多活或自动漂移。

---

## 启动失败与切换策略

| Option | Description | Selected |
|--------|-------------|----------|
| 严格失败，不自动重试 | 任一网络准备或校验失败就直接失败 | ✓ |
| 同一绑定做有限重试 | 对同一出口和配置自动重试 1 到 2 次 | |
| 允许一次更重的恢复动作 | 自动清理网络状态并重建一次再决定成败 | |

**User's choice:** 严格失败，不自动重试
**Notes:** 失败必须清楚暴露，不允许悄悄换路或自动恢复。

---

## DNS 受控方式

| Option | Description | Selected |
|--------|-------------|----------|
| DNS 跟着该出口 IP 的隧道一起走 | 把出口与 DNS 视为同一路径契约 | ✓ |
| DNS 走统一的受控解析器，但仍在隧道内 | 共享解析器，但不一定与出口身份完全一致 | |
| 你来定具体 DNS 方案，但要求结果可验证 | 只锁死“不回落 + 可验证” | |

**User's choice:** DNS 跟着该出口 IP 的隧道一起走
**Notes:** 不接受与当前活跃出口路径分离的 DNS 方案。

---

## 就绪验证与留痕强度

| Option | Description | Selected |
|--------|-------------|----------|
| 基础强校验 | 验证出口 IP、DNS 路径和非隧道直连阻断，并写入任务/事件 | ✓ |
| 更严格的扩展校验 | 加入更多泄漏探测或多轮探测 | |
| 基础强校验 + 运行时持续标坏 | 启动前校验，同时运行中发现异常就标坏 | |

**User's choice:** 基础强校验
**Notes:** 用户随后又进一步强调，运行中一旦出口异常或出现旁路，也必须立刻把主机标记为失败，这一补充约束已写入 CONTEXT.md。

---

## 出口 IP 独占性

| Option | Description | Selected |
|--------|-------------|----------|
| 单时刻全局独占 | 一个出口 IP 任意时刻只能被一台主机活跃使用 | |
| 允许多主机共享同一个出口 IP | 多个主机可以同时绑定并使用同一个出口 IP | |
| 绑定可共享，但活跃使用必须独占 | 可预绑定多台，但同一时刻只有一台可实际占用 | |
| Other | 每台主机运行时只有一个活跃出口；同一个出口 IP 可以分配给多个账号/主机，并允许同时运行共用该 IP | ✓ |

**User's choice:** 每台主机运行时只有一个活跃出口；同一个出口 IP 可以分配给多个账号/主机，并允许同时运行共用该 IP
**Notes:** 用户补充了业务语义：每个主机背后绑定一个独立的 `claude` 账号，唯一性在账号层，不在 IP 资源层。

## 运行中隧道异常后的主机状态

| Option | Description | Selected |
|--------|-------------|----------|
| 立刻标异常并阻止后续接入 | 发现隧道/DNS/旁路异常就把主机打坏 | |
| 只记事件，不自动打坏主机 | 先记录异常，主机状态不变 | |
| 先记异常，再给一次短暂恢复窗口 | 观测后再决定是否标坏 | |
| Other | 绝不允许存在未配置出口；一旦当前 IP 挂掉或变化，主机立刻标记为失败，直到原出口恢复并重新校验通过 | ✓ |

**User's choice:** 绝不允许存在未配置出口；一旦当前 IP 挂掉或变化，主机立刻标记为失败，直到原出口恢复并重新校验通过
**Notes:** 用户明确说明这是第一优先级，因为出口变化会触发 `claude` 账号风控。

## 网络准备时机

| Option | Description | Selected |
|--------|-------------|----------|
| 创建时就接好网络并完成强校验 | 容器一创建出来就具备受控出口能力 | |
| 启动时再接网络并做强校验 | 只有真正运行前才拉起隧道和校验 | ✓ |
| 创建时做静态绑定，启动时做最终强校验 | 先准备元数据，运行前再做最终接线 | |

**User's choice:** 启动时再接网络并做强校验
**Notes:** 主机进入可运行态必须与“出口正确且未泄漏”严格绑定。

## 运维可见性

| Option | Description | Selected |
|--------|-------------|----------|
| 按失败类型细分记录 | 至少区分绑定缺失、IP 不匹配、DNS 不受控、阻断失效、出口不可达 | ✓ |
| 保留简短摘要即可 | 统一网络失败摘要，细节看日志 | |
| 细分记录 + 明确高危级别 | 在细分基础上再做更强等级模型 | |

**User's choice:** 按失败类型细分记录
**Notes:** 先把失败原因拆开记录，不额外引入更复杂的危急等级模型。
