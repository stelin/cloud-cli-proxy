**Step 2: 创建 tests/smoke/test_helper/common.bash**

提供以下辅助函数：
```bash
# get_free_port: 获取一个空闲端口
get_free_port() {
  python3 -c 'import socket; s=socket.socket(); s.bind(("",0)); print(s.getsockname()[1]); s.close()'
}

# start_mock_server PORT STATUS_CODE BODY: 在指定端口启动一个返回固定响应的 HTTP mock
# 使用 python3 -m http.server 的自定义版本或 bash + socat 方式
# 把 PID 存到 MOCK_PID 变量供 teardown 使用
start_mock_server() {
  local port="$1" status="$2" body="$3"
  # 使用 python3 创建简易 HTTP server
  python3 -c "
import http.server, sys, threading
class H(http.server.BaseHTTPRequestHandler):
    def do_POST(self): self.send_response($status); self.send_header('Content-Type','application/json'); self.end_headers(); self.wfile.write(b'$body')
    def do_GET(self): self.do_POST()
    def log_message(self,*a): pass
s = http.server.HTTPServer(('127.0.0.1',$port), H)
s.serve_forever()
" &
  MOCK_PID=$!
  sleep 0.3  # 等待 server 就绪
}

# kill_mock_server: 清理 mock server 进程
kill_mock_server() {
  if [ -n "${MOCK_PID:-}" ]; then
    kill "$MOCK_PID" 2>/dev/null || true
    wait "$MOCK_PID" 2>/dev/null || true
  fi
}
```

2. `@test "account_disabled → exit 11"` — mock 返回 403 + account_disabled，验证 exit 11
3. `@test "account_expired → exit 12"` — mock 返回 403 + account_expired，验证 exit 12
4. `@test "host_not_found → exit 13"` — mock 返回 404 + host_not_found，验证 exit 13
5. `@test "connection failure → exit 2"` — 不启动 mock server，curl 失败，验证 exit 2 且 output 包含 "连接控制面失败"
6. `@test "unknown error_code → exit 1"` — mock 返回 400 + `{"error_code":"unknown_xyz","message":"未知"}`, 验证 exit 1

注意：bootstrap 脚本使用 `read -r username` 从 stdin 读取，测试需要用 `echo -e 'user\npass' | bash deploy/bootstrap/cloud-bootstrap.sh` 方式提供输入。由于脚本有 `--fail` curl flag，非 2xx 响应会触发 curl 失败而不是正常解析 JSON。检查脚本实际行为：curl 的 `--fail` 在 4xx/5xx 时使 curl 返回非零退出码，但 `2>&1` 会捕获错误。脚本在 curl 失败时进入 `|| { exit 2 }` 分支。所以对于 401/403 响应，脚本实际上会走到 `exit 2`（连接失败），而不是解析 error_code。

**重新审查脚本逻辑：**
脚本中 `curl --fail` 对 4xx 返回码会触发 curl 非零退出。但脚本写法是 `response=$(curl --fail ... 2>&1) || { exit 2 }`。这意味着 401 响应不会被解析到 error_code，而是直接 exit 2。

但这不对——仔细看：`--fail --show-error --silent` 的 curl 在 HTTP 4xx 时返回非零，`2>&1` 把 stderr 重定向到 stdout，response 会是空字符串（curl --fail 不输出 body），然后进入 `|| { exit 2 }` 分支。

所以当前的 bootstrap 脚本对于 401/403/404 等响应，实际行为是统一 exit 2（连接失败）。错误码解析路径只在 curl 成功（HTTP 200）但响应体包含 error_code 时才触发。

这意味着 BATS 测试需要根据脚本**实际行为**编写：
- HTTP 200 + body 包含 error_code → 解析 error_code 并按 case 退出
- HTTP 4xx/5xx → curl --fail 触发非零 → exit 2
- 连接失败（server 不存在）→ curl 非零 → exit 2
