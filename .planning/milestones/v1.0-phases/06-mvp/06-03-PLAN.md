**docs/operations-manual.md 结构：**

```markdown
# Cloud CLI Proxy 运维手册

## 用户管理
### 创建用户
```bash
curl -X POST http://127.0.0.1:8080/v1/admin/users \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"username":"newuser","password":"初始密码"}'
```
### 禁用/启用用户
PATCH /v1/admin/users/{userID} body: {"status":"disabled"/"active"}
### 删除用户
DELETE /v1/admin/users/{userID}（有运行中主机时会拒绝）
### 密码轮换
POST /v1/admin/users/{userID}/rotate-password
### 设置到期时间
PUT /v1/admin/users/{userID}/expiry body: {"expires_at":"2026-12-31T23:59:59Z"}

## 出口 IP 管理
### 创建出口 IP
POST /v1/admin/egress-ips（包含 WireGuard 配置参数）
### 绑定到主机
POST /v1/admin/bindings body: {"host_id":"...","egress_ip_id":"..."}
### 解除绑定
DELETE /v1/admin/bindings/{bindingID}

## 主机运维
### 查看主机状态
GET /v1/admin/hosts
### 启动/停止/重建
POST /v1/admin/hosts/{hostID}/start|stop|rebuild

## 事件查看
### 查看最近事件
GET /v1/admin/events?limit=50
### 按用户筛选
GET /v1/admin/events?user_id={userID}
### 按时间范围
GET /v1/admin/events?since=2026-03-01T00:00:00Z&until=2026-03-31T23:59:59Z

## 备份与恢复
### 数据库备份
使用 deploy/scripts/backup.sh 执行 PostgreSQL dump
### 数据库恢复
`pg_restore -d cloudproxy backup.dump`

## 证书与密钥轮换
### JWT Secret 轮换
1. 生成新密钥
2. 更新 /etc/cloud-cli-proxy/env
3. 重启控制面

### WireGuard 密钥轮换
1. 在出口 IP 提供商处更新密钥
2. 通过 Admin API 更新出口 IP 的 WireGuard 配置
3. 重启受影响的主机

## 日志
### 查看控制面日志
`journalctl -u cloud-cli-proxy-control-plane -f`
### 查看 host-agent 日志
`journalctl -u cloud-cli-proxy-host-agent -f`
### 切换日志级别
设置 LOG_LEVEL=DEBUG 并重启服务
```

```markdown
# Cloud CLI Proxy 故障排查与恢复

## 常见故障

### 1. 控制面无法启动
症状: systemd 服务 failed
排查:
1. `journalctl -u cloud-cli-proxy-control-plane --no-pager -n 50`
2. 检查 DATABASE_URL 是否正确
3. 检查 PostgreSQL 是否运行 `systemctl status postgresql`
4. 检查端口是否被占用 `ss -tlnp | grep 8080`

### 2. 用户无法登录
症状: bootstrap 脚本返回 "连接控制面失败" (exit 2)
排查:
1. 确认控制面在运行 `curl http://127.0.0.1:8080/healthz`
2. 检查用户状态（active/disabled/expired）
3. 检查网络连通性

### 3. 主机启动失败
排查:
1. 查看任务状态 GET /v1/admin/tasks
2. 查看相关事件 GET /v1/admin/events?host_id=XXX
3. 检查 Docker daemon `docker info`
4. 检查 WireGuard 配置 `wg show`

### 4. 网络校验失败
排查:
1. 检查出口 IP 绑定状态
2. 检查 WireGuard 接口 `wg show`
3. 检查 nftables 规则 `nft list ruleset`
4. 手动校验 `nsenter --net=/var/run/netns/... curl ...`

### 5. 到期扫描未触发
排查:
1. 检查控制面日志中 expiry-scan job
2. 确认用户 expires_at 字段已设置且已过期
3. 手动触发对账 — 重启控制面

### 6. 数据库连接耗尽
排查:
1. `psql -c "SELECT count(*) FROM pg_stat_activity WHERE datname='cloudproxy'"`
2. 检查连接池配置
3. 重启控制面释放连接

## 灾难恢复（附录）
### 完全恢复流程
1. 在新宿主机上运行 deploy/scripts/deploy.sh
2. 从备份恢复 PostgreSQL
3. 重建受管镜像
4. 重新配置 WireGuard 和出口 IP
5. 验证 bootstrap 流程
```

**deploy/scripts/backup.sh：**
```bash
#!/usr/bin/env bash
set -euo pipefail
BACKUP_DIR="${BACKUP_DIR:-/var/backups/cloud-cli-proxy}"
RETENTION_DAYS="${RETENTION_DAYS:-7}"
DB_NAME="${DB_NAME:-cloudproxy}"
TIMESTAMP=$(date +%Y%m%d_%H%M%S)
BACKUP_FILE="${BACKUP_DIR}/${DB_NAME}_${TIMESTAMP}.dump"

mkdir -p "$BACKUP_DIR"
pg_dump -Fc "$DB_NAME" > "$BACKUP_FILE"
echo "Backup saved: $BACKUP_FILE ($(du -h "$BACKUP_FILE" | cut -f1))"

# 清理过期备份
find "$BACKUP_DIR" -name "${DB_NAME}_*.dump" -mtime +$RETENTION_DAYS -delete
echo "Cleaned backups older than ${RETENTION_DAYS} days"
```

脚本应使用 `pg_dump -Fc`（custom format，支持 pg_restore 选择性恢复）。
