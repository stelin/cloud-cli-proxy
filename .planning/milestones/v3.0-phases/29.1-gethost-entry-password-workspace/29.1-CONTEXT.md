# Phase 29.1 — Context

## 触发场景（线上 P0）

用户：`ssh ei2568@cloud-cli-proxy.zaneliu.me -p 2222`，输入后台显示的 8 位初始密码后报：

```
Failed to connect to container: ssh: handshake failed: ssh: unable to authenticate,
attempted methods [none password publickey], no supported methods remain
```

宿主诊断结果（host_id=`fc882ad1-d0bc-4c52-b9b6-9cb4f894528c`）：

```
docker exec $CT env | grep CONTAINER_
CONTAINER_USER=workspace
CONTAINER_SSH_PASSWORD=workspace          ← ❌ 应是 DB 中的 8 位 entry_password

docker logs $CT | grep password
Failed password for workspace from 10.99.23.4 port ... ssh2   ← proxy 用真实密码登录被拒
```

后台显示和数据库 `hosts.entry_password` 都是正确的随机密码，**只有容器内的 `workspace` 用户密码被 fallback 成了 `"workspace"`**。补登入站公钥也无效（同样走 `resolveTarget()` 链路，最终也用 `auth.EntryPassword` 做 password 子方法）。

## 根因

`internal/store/repository/queries.go::GetHost` 的 SELECT 列表**漏了 `entry_password`** 列：

```go
// 当前实现 — entry_password 不在 SELECT 里
SELECT id::text, user_id::text, status, COALESCE(short_id, ''),
       template_image_ref, home_volume_name, slot_key, timezone,
       hostname, memory_limit_mb, cpu_limit, disk_limit_gb,
       created_at, updated_at
FROM hosts WHERE id = $1
```

下游链路全部受影响：

1. `runtime/runtime_service.go:143` — `EntryPassword: host.EntryPassword` → 永远 `""`
2. `runtime/tasks/worker.go:167` — `firstNonEmpty(request.EntryPassword, "workspace")` 把 `CONTAINER_SSH_PASSWORD` 注入成 `"workspace"`
3. `runtime/tasks/worker.go:432` — `syncContainerCredentials` 同样 fallback 成 `"workspace"`，对运行中容器执行 `chpasswd workspace:workspace`
4. `controlplane/http/admin_hosts.go::Create` 把生成的 entry_password 写入 DB 并直接 return 给前端 → 后台显示是对的，但容器永远不一致

公钥路径同样断掉，因为 `ResolveContainerByPublicKey → resolveTarget` 返回的 `Password` 字段也是 `auth.EntryPassword`，proxy 在 `handleChannel` 里依次尝试 `Password / KeyboardInteractive / PublicKeys(s.hostKey)`，前两者用的密码与容器内不一致直接失败；最后一个能否成功取决于 proxy 自己的 host pubkey 是否被 `injectSSHKeys` 写进容器 `/workspace/.ssh/authorized_keys`（待 phase 内验证）。

## 影响面

- **所有走 `GetHost` 读全字段的链路**：runtime worker create/start/rebuild、`RotateSSHPassword`、entry API（部分）、admin host detail（如果 detail 也走 GetHost）。
- **历史所有创建的容器**：`/etc/shadow` 里 `workspace` 都是 `"workspace"` 的 yescrypt 哈希，密码完全脱离数据库。
- **公钥登录**：因为 proxy 的 fallback 链同样依赖 password，行为退化为只剩"proxy host key 是否在容器 authorized_keys"这一条路。

## 修复范围（草案，待 plan-phase 细化）

1. **核心修复**：`GetHost` SQL 加 `entry_password`，Scan 加 `&item.EntryPassword`。
2. **静默 fallback 改成 fail-fast**：
   - `worker.buildCreateArgs` / `syncContainerCredentials` 检测到 `request.EntryPassword == ""` 时，把任务标记失败 + RecordEvent error 级别（type: `runtime.entry_password_missing`），而不是替换成 `"workspace"`。
   - `entrypoint.sh` chpasswd 后加自检 `passwd -S "$RUN_USER" | awk '{print $2}' | grep -qE 'P|PS'`，否则 `exit 1` 让容器失败而不是起一个登不进去的"成功"容器。
3. **存量容器修复**：写一个一次性脚本/admin 接口，按 DB `hosts.entry_password` 批量 `docker exec chpasswd` 同步所有 running 容器；记 audit event。
4. **回归测试**：
   - 仓储层单测：`GetHost` 必须返回 `EntryPassword` 字段（针对当前 SQL 的字段顺序写一条断言式测试）。
   - worker 单测：`request.EntryPassword == ""` 时 `buildCreateArgs` / `syncContainerCredentials` 返回错误。
   - 端到端：新建主机 → 用 API 返回的 entry_password 做 SSH 登录 → 应成功。

## 立即缓解（已知方案，phase 内不要求 commit）

```bash
HOST_ID=<host-id>
PASS=$(docker compose exec -T postgres \
  psql -U cloudproxy -d cloudproxy -At \
  -c "SELECT entry_password FROM hosts WHERE id='$HOST_ID'")
echo "workspace:$PASS" | docker exec -i cloudproxy-$HOST_ID chpasswd
```
