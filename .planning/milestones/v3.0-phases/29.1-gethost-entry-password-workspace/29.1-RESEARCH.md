## User Constraints（摘自 29.1-CONTEXT.md）

### Locked Decisions

- 根因：`internal/store/repository/queries.go::GetHost` SELECT 列漏 `entry_password`，下游 `host.EntryPassword` 永远空串。

## Executive Summary

1. **根因代码原文已锁定**：`queries.go::GetHost`（第 687-713 行）SELECT 14 列，漏 `entry_password`；`Host` 结构体（`models.go:34-50`）已有 `EntryPassword string` 字段在第 5 位（紧随 `ShortID` 之后）。
2. **同类漏列函数共 5 处** — 不只是 `GetHost`。任何一处回归都会让对应调用方退化到空密码：`ListHosts` / `ListHostsByUserID` / `ListHostsWithUsername` / `ListRunningHosts` / `ListRunningHostsByUserID`。修复时必须整批补齐，否则留坑。
3. **静默 fallback 只有两处硬编码** `"workspace"` 字面量（`worker.go:167` 和 `worker.go:432`），以及 `entrypoint.sh:124` 的 `${CONTAINER_SSH_PASSWORD:-workspace}`。定位精确，diff 极小。
4. **Fail-fast 改造不会误伤历史容器**：`admin_hosts.go::Create` 生成密码早于 `UpsertHost`（`generateEntryPassword()` 必定返回 8 位字符串），且 `UpsertHost` RETURNING 已正确回写。DB 里所有历史 host 的 `entry_password` 非空——验证见 §7 风险 1。
5. **批量修复模式现成可复用**：`admin_hosts.go::syncContainerPassword`（第 916 行，包级 helper）已经是 `docker exec chpasswd` 的标准实现且**没有** `"workspace"` fallback，是本次批量 resync 的首选复用点；`ListRunningHosts` 已存在于 repo。
6. **测试模板已经在仓库里**：`migration_0014_test.go` 展示了用"把 SQL 提升为 package-level const + `strings.Contains` 断言"保护仓储层的模式；`ssh_inject_test.go` 展示了 `execInContainer` 包级变量替换 + `fakeWorkerRepo` 的 worker 单测模式。planner 直接模仿即可，无需引入 testcontainers。
7. **仓库无 DB 集成测试基础设施**：所有 `_test.go` 都是 unit + stub 级别，没有任何 testcontainers / dockertest / postgres fixture。端到端 "创建主机 → SSH 登录" 验证**只能手工**进行；不要让 planner 在本阶段引入新的测试框架。

**Primary recommendation:** 一次 commit 完成 SQL 侧所有 `Host` 读路径的 `entry_password` 补齐（含 Scan 顺序）+ 提取 SQL 为 package-level const 便于断言；第二次 commit 把 worker 两处 fallback 改成 fail-fast + `runtime.entry_password_missing` 事件；第三次 commit 加 `entrypoint.sh` 自检；第四次 commit 加批量 resync admin 端点。测试穿插在各 commit 内。

## 1. 仓储层现状

### 1.1 `GetHost` 原文（根因，`internal/store/repository/queries.go` 第 687-713 行）

```go
func (r *Repository) GetHost(ctx context.Context, hostID string) (Host, error) {
    var item Host
    if err := r.db.QueryRow(ctx, `
        SELECT id::text, user_id::text, status, COALESCE(short_id, ''), template_image_ref, home_volume_name, slot_key, timezone, hostname, memory_limit_mb, cpu_limit, disk_limit_gb, created_at, updated_at
        FROM hosts
        WHERE id = $1
    `, hostID).Scan(
        &item.ID,
        &item.UserID,
        &item.Status,
        &item.ShortID,
        &item.TemplateImageRef,   // ← Scan 顺序在此跳过了 EntryPassword
        &item.HomeVolumeName,
        &item.SlotKey,
        &item.Timezone,
        &item.Hostname,
        &item.MemoryLimitMB,
        &item.CPULimit,
        &item.DiskLimitGB,
        &item.CreatedAt,
        &item.UpdatedAt,
    ); err != nil {
        return Host{}, fmt.Errorf("get host: %w", err)
    }
    return item, nil
}
```

> 最小修复形态（不作为最终实现，仅示意 diff 大小）：
> SELECT 在 `COALESCE(short_id, '')` 后插入 `COALESCE(entry_password, '')`，Scan 对应在 `&item.ShortID` 后插入 `&item.EntryPassword`。

## 2. 消费方调用链（下游）

### 2.1 `internal/runtime/runtime_service.go::QueueHostAction`

- 第 143 行：`EntryPassword: host.EntryPassword`（来自 `GetHost`）。修好 GetHost 后自动正确。
- 第 149-152 行：**已经存在** `slog.Warn("host entry_password is empty...")`——planner 任务之一是把这里升级成"直接 return error，不入队 + RecordEvent"。

### 2.2 `internal/runtime/tasks/worker.go`

两处 fallback 是全部硬编码 `"workspace"`：

| 位置 | 行 | 当前代码 | 修复方向 |
|------|----|----------|----------|
| `buildCreateArgs` | 167 | `"CONTAINER_SSH_PASSWORD="+firstNonEmpty(request.EntryPassword, "workspace")` | `EntryPassword == ""` → 返回 `error`，`createHost` 走现有 err-path（`w.repo.UpdateHostStatus(ctx, request.HostID, "failed")` + 返回 `taskStateFailed` 已在第 89-96 行就绪） |
| `syncContainerCredentials` | 431-432 | `user := firstNonEmpty(request.DefaultUser, "workspace")` + `pass := firstNonEmpty(request.EntryPassword, "workspace")` | `request.EntryPassword == ""` → RecordEvent(error, `runtime.entry_password_missing`) + 直接 return（**不能**真的执行 chpasswd，否则写入 `workspace`）。`user` 回退 `"workspace"` 仍然合理——那是 Linux 用户名不是密码，不受本 bug 影响。 |

> ⚠️ 注意 `buildCreateArgs` 第 160 行的 `linuxUser := firstNonEmpty(request.DefaultUser, "workspace")` 是 Linux 用户名 fallback，**保留**。本阶段只清洗密码语义的 fallback。

### 2.3 `internal/controlplane/http/admin_hosts.go::RotateSSHPassword`（第 368-418 行）

- 现有流程：`GetHost → generateEntryPassword() → UpdateHostEntryPassword → (if running) syncContainerPassword(container, "workspace", newPassword)`。
- **独立于本 bug**：`RotateSSHPassword` 自己新生成密码，不依赖 GetHost 读出的 `host.EntryPassword`，所以现有用户执行该接口不会踩 fallback。
- 但 `syncContainerPassword`（第 916 行，包级函数）恰好是**没有** `"workspace"` fallback 的干净实现——是本阶段批量 resync 的首选复用点（§5）。

### 2.4 `internal/sshproxy/*`

- `resolver.go::resolveTarget`（第 78-96 行）返回的 `ContainerTarget.Password = auth.EntryPassword`，来源是 `GetHostByShortID → HostSSHAuth`，**该 SQL 已含 `entry_password`**（`getHostByShortIDSQL`，第 1154 行）。sshproxy 读到的密码本身是对的。
- `proxy.go::handleChannel`（第 212-235 行）auth 三元组：`ssh.Password(pass) / passwordKeyboardInteractive(pass) / ssh.PublicKeys(s.hostKey)`。前两个依赖与容器内 `/etc/shadow` 一致的密码——本阶段 bug 修复完成后自然通。
- 第 225-228 行有一层额外 fallback：`pass := targetPassword; if pass == "" { pass = s.containerPassword }`。`s.containerPassword` 在 `NewServer`（第 37-48 行）又用 `"workspace"` 兜底——这是 proxy 启动期的全局默认值，不是每连接 fallback。**本阶段不必动**，但要在 REVIEW 时确认该全局默认不会在任何路径被单 host 命中（`targetPassword` 来自 DB，修复后恒不为空串）。

**结论：** sshproxy 链路不需要本阶段代码修改，只需在验证步骤确认 `targetPassword != ""`。

## 3. 镜像侧（`deploy/docker/managed-user/entrypoint.sh`）

- 第 123-124 行：
  ```bash
  CONTAINER_USER="${CONTAINER_USER:-workspace}"
  CONTAINER_PASSWORD="${CONTAINER_SSH_PASSWORD:-workspace}"
  ```
- 第 150 行：`echo "${RUN_USER}:${CONTAINER_PASSWORD}" | chpasswd`

**修复点**：紧接第 150 行之后（第 151 行开始）加自检，失败即 `exit 1`：

```bash
# 验证密码已被成功设置，避免历史"密码退化为 workspace"的静默失败复现。
# passwd -S 第 2 列：P / PS = 正常设置密码，L / LK = 锁定，NP = 无密码。
status="$(passwd -S "${RUN_USER}" 2>/dev/null | awk '{print $2}' || echo UNSET)"
case "$status" in
  P|PS) ;;
  *)
    echo "[entrypoint] FATAL: password status for ${RUN_USER} is '${status}', refusing to start" >&2
    exit 1
    ;;
esac
```

**不改范围**：其他段（`write_desktop_config`、`prepare_v3_*`、KasmVNC）保持原样，避免污染提交范围。

**兼容性**：历史上 `CONTAINER_SSH_PASSWORD` 确实可能为空（老 worker 就是这么进来的），所以新镜像不能破坏"空密码+工作 shell"的幂等；但既然本阶段要求 fail-fast，空密码就应该让容器直接启动失败——这与 CONTEXT.md 锁定意图一致。

## 7. 风险与开放问题

### 7.1【低】fail-fast 是否误伤历史运行容器？

- **验证路径**：`admin_hosts.go::Create` 第 174 行 `hostEntryPassword := generateEntryPassword()`（该函数返回 8 位字符串，`admin_users.go:106`），随后 `UpsertHost(EntryPassword: hostEntryPassword, ...)`（第 194 行）。因此 DB 中所有 host 的 `entry_password` 都非空。
- **剩余疑点**：是否有 host 通过 migration seed / 手工 SQL 插入且绕过 `generateEntryPassword()`？查 `internal/store/migrations/*.sql` 与已知 0001-0014 文件，未见直接 INSERT hosts 的 seed。
- **缓解**：批量 resync 端点首轮执行时，对 `EntryPassword == ""` 的 host 记 `runtime.entry_password_missing` 事件而非 chpasswd，给运维一次人工补票窗口。

### 7.2【低】`host.EntryPassword == ""` 是否被某处当成"无密码"的语义？

- 完整 grep `EntryPassword` 在代码（非测试、非计划）出现位置：`runtime_service.go:143,149` / `worker.go:167,432` / `admin_hosts.go:115,174,194,248,253` / `sshproxy/resolver.go:41,44,94` / `admin_hosts.go:64`（List 时强制置空）。
- 没有一处把 `== ""` 当 "未配置" 合法语义——要么是"准备序列化给前端（用 `omitempty` 抑制）"，要么是"本应有值"。
- **结论**：语义从"fallback 成 workspace"整体翻到"空=异常"是安全的。

### 7.3【中】`GetHostDetail` 返回体是否会新泄漏 `entry_password`？

- admin `Get()` 已显式 `resp.Host.EntryPassword = ""`（`admin_hosts.go:115`）；`List()` 已 `hosts[i].EntryPassword = ""`（第 64 行）。
- **需要 planner 验证**：用户面板 handler（`entry.go` / `ssh_keys.go` 等非 admin 链路）是否有读 `GetHostDetail` 或 `ListHosts*` 后直接 `json.NewEncoder(w).Encode(...)` 的地方。**建议 planner 在 PLAN 里列一条 "grep 全部 Host JSON 响应路径确认 EntryPassword 被屏蔽" 的任务**。
- 缓解：`Host` 结构体字段已是 `omitempty` + 小写 `entry_password`，默认不暴露空串，但非空时会直接出现在响应里。

### 7.4【低】`syncContainerPassword` 当前不可 mock

- `admin_hosts.go:916` 是 package-private function，不是 package-level `var`。批量 resync 端点若想做 HTTP 单测，需要把它提升为 `var syncContainerPassword = func(...)` 以便测试替换——与 `worker.go:673` 的 `execInContainer` 模式一致。
- **建议 planner 在 PLAN 里单列一小步 "把 `syncContainerPassword` 改为 var 允许 mock"，不扩大改动语义**。

### 7.5【开放】公钥链路是否也被同一 Bug 间接影响？

- CONTEXT.md 已把 "公钥路径修复" 列为 OUT OF SCOPE，这里只登记观察：
  - `sshproxy/proxy.go` 的第三种 auth `ssh.PublicKeys(s.hostKey)` 是 **proxy 自己的 hostKey**，依赖 `loadProxyPublicKey()` 把 proxy 的 `ssh_host_ed25519_key.pub` 注入容器 `/workspace/.ssh/authorized_keys`（`worker.go:460-462`）。
  - 若 `DATA_DIR/ssh_host_ed25519_key.pub` 不存在，第三种 auth 也会失败——这是 deployment 级问题，不在代码修复面。
- **给 planner 的指令**：PLAN 可以加一条 "Manual UAT：验证 `$DATA_DIR/ssh_host_ed25519_key.pub` 存在"的检查项，**不要**把 `injectSSHKeys` 重构拉进来。
