```go
cleanup = func() {
    if promoterCancel != nil {
        promoterCancel()
    }
    if promoter != nil {
        promoter.Wait()
```

**重要：** promoter stats flush 必须在 `writeLastSessionWarn` 之前。当前代码 `writeLastSessionWarn` 在 `MountWorkspace` 的成功分支调用（L280），cleanup 闭包内只做资源回收，不写 last-session。因此 stats flush 需要移到 cleanup 闭包之外——或者改在 promotion 计数更新时直接写 `snapshot` 成员变量。

**推荐方案：** 把 `snapshot` 指针传递进 cleanup 闭包，stats flush 在 cleanup 内完成（因为 cleanup 被执行时已经是最新的 stats）。但当前 `snapshot` 是局部变量，闭包可以捕获指针。具体实现：

```go
// promotion stats flush 放入 cleanup 闭包（在 promoter.Wait() 之后）
// writeLastSessionWarn 在 MountWorkspace 成功分支调用，此时 snapshot 已包含 promotion 字段
```

实际上 `writeLastSessionWarn` 在 `MountWorkspace` 成功分支 L280 调用，而 cleanup 在 `finalCleanup` 中执行（在 `writeLastSessionWarn` **之前**还是**之后**？）

阅读代码 L274-290：`printBanner` → `writeLastSessionWarn` → `finalCleanup`（L283-289 构造）→ `return finalCleanup, mode, nil`。所以 `writeLastSessionWarn` 在 cleanup **执行之前**被调用。这意味着 snapshot 需要在 `writeLastSessionWarn` 之前就填充 promotion stats。但此时 promoter 还在运行中...

**正确方案：** promotion stats 在 cleanup 闭包内 flush，但不在 MountWorkspace 成功分支快照时写。改为在 cleanup 执行时写最后一次 snapshot（或者 promotion stats 在每次 `writeLastSessionWarn` 被调用时采用 "best-effort 当前值"）。

**简单方案（推荐）：** promoter.Stats() 在 `writeLastSessionWarn` 之前调用一次（获取当前值），即使 promoter 还在运行中，这只是统计的"快照"。实际语义正确——last-session.json 记录的是 mount 就绪时的状态快照，promotion 还在进行中，统计按当前值写入。用户可以在 cleanup 后查看最终的 last-session.json（cleanup 时不会再次写入）。若需要最终值，doctor 在读取时能接受"上次会话"的数据即可。

具体：在 `writeLastSessionWarn` 之前（L280 附近）插入：

```go
if promoter != nil {
    count, bytes, failed := promoter.Stats()
    snapshot.PromotionCount = count
    snapshot.PromotionBytes = bytes
    snapshot.PromotionFailedCount = failed
}
```

这符合 Phase 36 OversizedFiles 的模式——OversizedFiles 也是在 `writeLastSessionWarn` 之前从 `hotStatus` 赋值（L246）。
