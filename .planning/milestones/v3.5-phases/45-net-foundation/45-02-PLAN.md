<bind_mount_injection_strategy>
<!-- 关键设计决策：BindMount 注入点 -->

worker 容器创建在 internal/runtime/tasks/worker.go::buildCreateArgs（用 agentapi.HostActionRequest.BindMounts）。
ContainerProxyProvider.PrepareHost 在 worker 容器创建之后才执行（providers 链由 runtime 编排）。

但 sing-box gateway 必须先启动、tun0 必须可用，才能让 worker 内 DNS 查询通过 172.19.0.1 工作。
worker 容器创建时 bind mount 一旦生效，worker 内 DNS 就锁定到 172.19.0.1 —— 但此时 gateway 可能还没起来 → 这是 RESEARCH §Pitfall 2 的风险。

**本 plan 的处理方式**：
1. control-plane 在生成 HostActionRequest 之前，**先**调用 `writeContainerDNSConfig(hostID)` 写源文件到 `<gatewayConfigDir>/resolv.conf` 与 `<gatewayConfigDir>/nsswitch.conf`（这两条路径不依赖 gateway 容器是否启动，只是宿主文件系统操作）。
2. 在 HostActionRequest.BindMounts 里注入两条 ReadOnly bind mount。
3. worker 容器创建后，**默认会走 docker DNS（127.0.0.11）失败 → fallback** —— 不行；docker 容器创建时 resolv.conf 已被 bind mount 接管，没有 fallback。
4. 因此**必须在调用 worker.go::CreateHost 之前** PrepareHost 已经把 gateway 启动好。当前调用链确实是这个顺序：先 PrepareHost（gateway healthy + tun0 监听 172.19.0.1）→ 然后 worker connect to gateway network → 此时 worker 内 DNS 查 172.19.0.1 可达。

**调用链验证**（read-only，由 Task 1a 验证现状；Task 1b 落地拆分）：
- runtime/tasks/worker.go 在 docker create worker 之前/之后调用 PrepareHost？
- 如果 worker.go::createHost 先创建容器再调 PrepareHost，bind mount 会让 worker 容器 entrypoint 启动时（在 PrepareHost 之前）DNS 失败。

**Task 1a 必须做现状勘察**：grep worker.go 看 PrepareHost / NetworkProvider 调用与 dockerCreate / dockerStart 的相对顺序。
- 如果 PrepareHost 在 docker create 之前 → 安全，按计划注入 BindMount
- 如果 PrepareHost 在 docker start 之后但 docker create 之前 → 通常 worker 容器 entrypoint 在 docker start 时才执行，仍安全
- 如果 PrepareHost 在 docker start 之后 → 不安全，本 plan 需调整顺序（额外 task）

**注入点选择**：在 worker.go::buildCreateArgs 中直接 append 两条 `--mount type=bind,src=<gatewayConfigDir>/resolv.conf,dst=/etc/resolv.conf,readonly`；不依赖 HostActionRequest（避免 control-plane → agent 协议改造）。计算 `gatewayConfigDir(hostID)` 需要把 helper 提升或复制到 worker 包。**推荐**：worker.go 通过 `request.HostID` + 与 network 包共享的常量函数 `network.GatewayConfigDir(hostID)`（导出 helper）拼出路径。

**调用链顺序硬约束（B1+B2 修复要点）**：当前 `internal/runtime/tasks/worker.go::createHost`（line 351-377）的顺序是「buildCreateArgs → docker create → docker start → buildEgressConfig → PrepareHost」。这意味着 worker 容器在 sing-box gateway 启动之前就 run 起来，bind mount 接管的 /etc/resolv.conf 指向尚未监听的 172.19.0.1，第一次 DNS 查询必然失败。

**方案 A（本 plan 采纳）**：在 `ContainerProxyProvider` 引入新方法 `PrepareGateway(ctx context.Context, hostID string) error`，把 PrepareHost 中「create network + start gateway + 等 sing-box healthy + 写两个 rule-set placeholder + 写 resolv.conf/nsswitch.conf 源文件」全部剥离到 PrepareGateway；`PrepareHost` 缩减到「connect worker netns + configure routes」。worker.go::createHost 调整为：
```go
// 1. 准备 egress 配置（提前到 docker create 之前，用于 PrepareGateway）
egressCfg, err := w.buildEgressConfig(ctx, request.HostID)
if err != nil { return err }
if egressCfg != nil {
    // 2. 起 gateway + sing-box healthy + 写 placeholder + 写 DNS 源文件
    if err := w.provider.PrepareGateway(ctx, network.HostNetworkSpec{HostID: request.HostID, Egress: egressCfg}); err != nil {
        return fmt.Errorf("prepare gateway before create: %w", err)
    }
}
// 3. 此时 tun0 (172.19.0.1) 已监听，且 <DATA_DIR>/gateway/<hostID>/resolv.conf 已写盘
args, err := w.buildCreateArgs(request, containerName, hostname) // bind mount 路径生效
if err != nil { return err }
if err := w.runDocker(ctx, args...); err != nil { return err }
if err := w.runDocker(ctx, "start", containerName); err != nil { ... } // worker 起来时 DNS 已可达
if egressCfg != nil {
    // 4. 仅做 connect + routes
    if err := w.provider.PrepareHost(ctx, spec); err != nil { ... }
}
```

**Plan 01 引入的小写 `gatewayConfigDir(hostID)` 调用同步重写（B3）**：Plan 01 在 PrepareHost 内新增了若干小写 `gatewayConfigDir(hostID)` 调用（用于写两个 rule-set placeholder）。本 plan Wave 2 把 helper 改名为导出 `GatewayConfigDir(hostID)` 时，**必须同步把 Plan 01 引入的所有调用一起重写**；Task 1b 的 acceptance criteria 含 `grep -rn "gatewayConfigDir\b" internal/` 返回 0 行（小写已绝迹）。
