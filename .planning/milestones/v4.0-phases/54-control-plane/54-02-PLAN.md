# Plan 54-02: sing-box 容器内 config 生成 + host-agent 注入

## Objective

实现 Plan 54-01 留下的 `writeContainerSingBoxConfig` 真实逻辑：

1. 基于 `EgressConfig.Proxy.OutboundConfig`（用户配置的 sing-box outbound JSON）+ 上游代理 server IP 解析结果，构建**容器内自跑**的 sing-box config（与 v3.5 gateway 容器跑的 `buildGatewaySingBoxConfig` 不同 — 后者跑在独立容器、address 是 `172.19.0.1/30` 隔离子网；前者跑在 user 容器自身 netns，无外部隔离子网，address 仍可保留 `172.19.0.1/30`，但 DNS / route / outbound 语义需对齐 Phase 53 entrypoint 的 `apply_nft_or_die` / `lock_resolv_conf` 约束）。
2. 把生成的 config JSON 写入 `<DATA_DIR>/gateway/<hostID>/config.json`，文件权限 `root:singbox 0640`（uid=0, gid=9000, mode=0o640），与 Phase 53 entrypoint `start_singbox_or_die` 的 hard-assert 严格对齐。
3. 验证容器启动后 `docker exec ls /etc/sing-box/config.json` 失败（entrypoint `remove_singbox_config` 在 sing-box 加载后 shred / rm）。

### T1 — 新建 `container_singbox_config.go` 文件

文件：`internal/network/container_singbox_config.go`

文件骨架（不带 build tag，darwin 也要能编译以让 `internal/network` 单测在 macOS 跑通）：

```go
package network

import (
    "encoding/json"
    "fmt"
)

// buildContainerSingBoxConfig 渲染容器内自跑的 sing-box config JSON。
//
// v4.0 (Phase 54) 同容器架构下，sing-box 跑在 user 容器自身 netns 内：
//   - tun inbound 地址沿用 172.19.0.1/30（与 v3.5 gateway 兼容；address 是容器
//     私有 tun 接口 IP，不与 docker bridge 网段冲突）
//   - DNS 走 stub resolver（type=local + type=https + detour=proxy-out），与
//     v3.5 buildGatewayDNS 同款（dns-local 兜底 + dns-proxy via DoH 1.1.1.1）
//   - route default_interface = "eth0"（容器内 docker bridge 接口）
//   - route final = "proxy-out"
//   - outbound 含 proxy-out（用户配置）+ direct（兜底）
//
// 与 v3.5 buildGatewaySingBoxConfig 的差异：
//   - v3.5 gateway 容器作 sidecar，address/172.19.0.1/30 在隔离 bridge 网络下；
//     v4.0 user 容器自带 tun，无隔离 bridge，但 tun 接口仍可用同地址（仅容器内可见）
//   - v4.0 容器内 entrypoint lock_resolv_conf 已经把 /etc/resolv.conf 写死 127.0.0.1，
//     故 sing-box DNS 配置必须真的提供 stub resolver 监听 127.0.0.1:53
func buildContainerSingBoxConfig(outboundRaw json.RawMessage, dnsServer, proxyServerIP string) ([]byte, error) {
    proxyOut, err := buildGatewayProxyOutbound(outboundRaw, proxyServerIP)
    if err != nil {
        return nil, err
    }
    directOut, err := buildGatewayDirectOutbound()
    if err != nil {
        return nil, err
    }
    tunIn, err := buildContainerTunInbound()
    if err != nil {
        return nil, err
    }

    cfg := map[string]any{
        "log":      map[string]any{"level": "info"},
        "dns":      buildGatewayDNS(),
        "inbounds": []json.RawMessage{tunIn},
        "outbounds": []json.RawMessage{proxyOut, directOut},
        "route": map[string]any{
            "default_interface":       "eth0",
            "default_domain_resolver": map[string]any{"server": "dns-local"},
            "rules":                   buildContainerRouteRules(proxyServerIP),
            "final":                   "proxy-out",
        },
    }
    return json.MarshalIndent(cfg, "", "  ")
}

// buildContainerTunInbound 渲染容器内 tun inbound。
// 与 buildGatewayTunInbound 区别：v4.0 同容器架构下不需要 endpoint_independent_nat
// （流量从 user 进程 → tun0 → sing-box → eth0 是单向的，无回程 NAT），但保留
// strict_route + auto_route 与 v3.5 对齐；stack 仍 "system" 走内核网络栈，性能优。
func buildContainerTunInbound() (json.RawMessage, error) {
    raw, err := json.Marshal(map[string]any{
        "type":         "tun",
        "tag":          "tun-in",
        "address":      []string{"172.19.0.1/30"},
        "auto_route":   true,
        "strict_route": true,
        "stack":        "system",
        "sniff":        true,
    })
    if err != nil {
        return nil, fmt.Errorf("marshal container tun inbound: %w", err)
    }
    return raw, nil
}

// buildContainerRouteRules 渲染容器内 sing-box route.rules（精简版，无 v3.5
// 白名单 rule-set 路径 —— bypass 白名单是 v3.5 网络架构特有，v4.0 同容器先
// 不引入，待 v4.1 评估）。
//
// 4 条规则：
//  1. action=sniff（嗅探 tls/http/quic/dns）
//  2. protocol=dns + action=hijack-dns（让所有 DNS 流量经 sing-box DNS server）
//  3. ip_cidr=<proxy_ip>/32 → direct（避免 sing-box 回环访问自身代理服务器）
//  4. ip_is_private=true → direct（RFC1918 / CGNAT / 链路本地 / ULA）
// route.final=proxy-out 在调用方设置兜底。
func buildContainerRouteRules(proxyServerIP string) []map[string]any {
    return []map[string]any{
        {"action": "sniff", "sniffer": []string{"tls", "http", "quic", "dns"}},
        {"protocol": "dns", "action": "hijack-dns"},
        {"ip_cidr": []string{proxyServerIP + "/32"}, "action": "route", "outbound": "direct"},
        {"ip_is_private": true, "action": "route", "outbound": "direct"},
    }
}
```

**实现注意**：

- 复用 v3.5 `buildGatewayProxyOutbound` / `buildGatewayDirectOutbound` / `buildGatewayDNS`（来自 `gateway_singbox_config.go`，文件保留作 dead code 的导出符号供本 plan 复用，D-54-7 路径）。如果 Phase 55 / Phase 56 真要删 `gateway_singbox_config.go`，三个 helper 应迁移到 `container_singbox_config.go`。本 plan 不做迁移以减少 diff 面积。
- **不**包含 v3.5 `buildGatewayRouteRuleSet` 与 whitelist 文件路径引用 — bypass 白名单在 v4.0 单容器是单独议题，留 v4.1。
- DNS 走 stub resolver 127.0.0.1:53 = sing-box 默认行为（dns inbound 实际由 sing-box 自身监听，无需在 config 里显式声明端口）。entrypoint `lock_resolv_conf` 写死 nameserver 127.0.0.1 与之配套。

### T2 — 实装 `writeContainerSingBoxConfig` 真实逻辑

文件：`internal/network/container_proxy_provider.go`

替换 Plan 54-01 stub：

```go
// writeContainerSingBoxConfig 把生成的 sing-box config 写到 host 端
// SingBoxConfigDir(hostID)/config.json，权限严格 root:singbox 0640（D-54-2）。
//
// 严格权限对齐 Phase 53 entrypoint start_singbox_or_die hard-assert
// （deploy/docker/managed-user/entrypoint.sh L178-189）：
//   - perm == 0o640（owner read+write，group read，other 0）
//   - owner uid == 0（root）
//   - group gid == 9000（singbox，与 Dockerfile groupadd --gid 9000 singbox 对齐）
// 任一不匹配 entrypoint 立即 fail-closed (exit 1)，容器永远起不来。
//
// chown 失败仅 WARN 不 FATAL：darwin 本地开发环境 owner 切到 0/9000 可能因
// 非 root 跑 cmd/control-plane 而失败；prod Linux 控制面以 root systemd 跑
// 时必然成功。darwin 本地 host create 也无法真正跑通（无 /dev/net/tun），
// 失败语义统一交给 verifyWorkerNetwork 兜底。
func writeContainerSingBoxConfig(hostID string, egress *EgressConfig) error {
    if egress == nil || egress.Proxy == nil {
        return fmt.Errorf("writeContainerSingBoxConfig: nil egress / proxy")
    }
    serverIP, _, err := extractProxyServer(egress.Proxy.OutboundConfig)
    if err != nil {
        return fmt.Errorf("resolve proxy server: %w", err)
    }
    cfg, err := buildContainerSingBoxConfig(egress.Proxy.OutboundConfig, egress.Proxy.DNSServer, serverIP)
    if err != nil {
        return fmt.Errorf("build container sing-box config: %w", err)
    }
    dir := SingBoxConfigDir(hostID)
    if err := os.MkdirAll(dir, 0o755); err != nil {
        return fmt.Errorf("mkdir config dir: %w", err)
    }
    cfgPath := filepath.Join(dir, "config.json")
    // 先写 0o600 防止任何窗口期 group/world readable；再 chmod 到 0640；
    // 再 chown root:singbox。三步顺序：write(0600) → chown → chmod(0640)。
    if err := os.WriteFile(cfgPath, cfg, 0o600); err != nil {
        return fmt.Errorf("write config: %w", err)
    }
    if err := os.Chown(cfgPath, 0, singboxGroupGID); err != nil {
        // darwin / 非 root 跑测试时 chown 必失败，仅 WARN。Linux prod root 必成功。
        // 真正的安全保证由 entrypoint hard-assert 兜底（perm/owner/group 不对则 exit 1）。
        // 不再 return err，避免 darwin 单测 false-positive。
        // log.Printf 不在 free function 内做，由调用者 provider.logger.Warn 在外层记录。
        // 此处仅以错误形式带出，上层判断 errno。
        return fmt.Errorf("chown root:singbox: %w", err)
    }
    if err := os.Chmod(cfgPath, 0o640); err != nil {
        return fmt.Errorf("chmod 0640: %w", err)
    }
    return nil
}

// singboxGroupGID 与 Dockerfile groupadd --gid 9000 singbox 严格对齐（D-54-2）。
// 写死 numeric gid 避免依赖 host 上 getent group singbox（控制面可能没装 singbox 组）。
const singboxGroupGID = 9000
```

**chown 错误处理改进**：函数返回 `error`，调用方（`PrepareGateway`）在 darwin 开发环境收到 `chown: operation not permitted` 时应 `logger.Warn` 但不阻断 `PrepareGateway`。最干净做法是在 `PrepareGateway` 内捕获 errno：

```go
// PrepareGateway 调整：chown 失败时降级为 warning
if err := writeContainerSingBoxConfig(spec.HostID, spec.Egress); err != nil {
    // darwin 本地开发：非 root 进程 chown 必失败，降级 warning
    if isChownPermissionError(err) {
        p.logger.Warn("container-proxy: chown root:singbox failed (likely non-root dev env), entrypoint hard-assert will catch this on Linux",
            "host_id", spec.HostID, "error", err)
        // 仍然返回 nil，让后续 docker create 继续；entrypoint fail-closed 兜底
        return nil
    }
    return fmt.Errorf("sing-box: write config: %w", err)
}
```

`isChownPermissionError` 用 `errors.Is(err, fs.ErrPermission)` 判断；如果误判可降级为字符串匹配 `"chown"` + `"operation not permitted"` 子串。

// isChownPermissionError 判定错误是否为 chown EPERM（darwin 非 root 跑测试时常见）。
// 真实安全保证由 Phase 53 entrypoint hard-assert 兜底，darwin 开发环境放过去即可。
func isChownPermissionError(err error) bool {
    if err == nil {
        return false
    }
    s := err.Error()
    return strings.Contains(s, "chown") && (strings.Contains(s, "operation not permitted") ||
        strings.Contains(s, "permission denied"))
}

## Risks

| Risk | Mitigation |
|---|---|
| `singbox` GID 在某 host 系统上可能被占用导致 chown 写到错误账号 | numeric 9000 是 Phase 53 Dockerfile 强约束；entrypoint hard-assert group 名 = "singbox"（不是 GID），如果 host 上 9000 是其他组 → chown 成功但 entrypoint stat group 不是 "singbox" → fail-closed exit 1。**这是设计意图的安全失败**，host-agent 不需要修复 |
| darwin 开发环境 chown EPERM 导致 `PrepareGateway` 一律失败、host 永远 create 不出来 | T5 降级处理：darwin 非 root 跑 chown 失败时仅 logger.Warn，PrepareGateway 仍 return nil；docker create 会因 bind mount source 文件存在而 succeed，但 entrypoint 在 Linux 容器内 stat 时会因 owner/group 错失败 — darwin 本来就跑不通 host 创建，行为一致 |
| 容器内 sing-box config 与 Phase 53 entrypoint 期望的 tun 接口名 "tun0" 不一致 | sing-box 1.13 默认 tun interface 名是 tun0 当没显式 `interface_name` 时；entrypoint `ip link show tun0` 与默认匹配。本 plan 不显式声明 interface_name 即可 |
| dns 走 stub resolver 但 sing-box config 未声明 dns inbound 端口 → sing-box 不监听 127.0.0.1:53 → entrypoint lock_resolv_conf 后所有 DNS 查询都失败 | sing-box `dns` 模块默认会接管 hijack-dns route 规则，无需单独 inbound。验证：本 plan T3 单测 `DNSStubServers` 锁双 server 存在 + entrypoint smoke deferred-to-Phase-55-CI 端到端验证 |
