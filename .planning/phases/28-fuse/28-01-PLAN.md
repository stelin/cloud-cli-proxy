**Shebang 与严格模式**（与 host-preflight.sh 一致）：

#!/usr/bin/env bash

```

**全局变量与计数器**：

PASS_COUNT=0
FAIL_COUNT=0
WARN_COUNT=0
CONTAINER_NAME="fuse-verify-$$"
IMAGE_NAME=""
```

**工具函数**（per D-09 结构化输出）：
- `pass(msg)` — 打印 `[PASS]  $msg` 并递增 PASS_COUNT
- `fail(msg)` — 打印 `[FAIL]  $msg` 并递增 FAIL_COUNT
- `warn(msg)` — 打印 `[WARN]  $msg` 并递增 WARN_COUNT
- `info(msg)` — 打印 `[INFO]  $msg`
- `cleanup()` — `docker rm -f "$CONTAINER_NAME" 2>/dev/null || true`，用 `trap cleanup EXIT`

**镜像名读取**（与 verify-managed-image.sh 一致）：
```bash
IMAGE_NAME="$(awk -F': ' '$1 == "local_dev_image_name" { print $2 }' deploy/docker/managed-user/image.lock)"
```
如果 IMAGE_NAME 为空则 fail 并退出。

**阶段 1：宿主机安全模块检测**（per D-05）：
- 检测 FUSE 内核模块：`modprobe fuse 2>/dev/null && test -c /dev/fuse`，PASS 则打印 "FUSE 内核模块: 已加载"，FAIL 则打印 "FUSE 内核模块: 未加载，请运行 modprobe fuse"
- 检测 AppArmor 状态：调用 `aa-enabled --quiet 2>/dev/null` 判断是否启用，打印状态（已启用/enforcing、未启用、未安装）
- 检测 fusermount3 AppArmor profile（per RESEARCH Pitfall 2）：`aa-status 2>/dev/null | grep -q "fusermount3"` 判断是否加载，加载则 WARN 提示 Ubuntu 25.04+
- 检测 Docker 安全选项：`docker info --format '{{.SecurityOptions}}'` 打印当前 Docker 安全模块配置
- 打印宿主机 OS 版本：读取 `/etc/os-release` 中的 PRETTY_NAME

**阶段 2：容器内真实 sshfs FUSE 挂载测试**（per D-02 SC-1，对齐 PATTERNS.md 中 mount.go 的 `mountpoint -q` 判据）：
- 使用 `docker run -d --name "$CONTAINER_NAME" --cap-add SYS_ADMIN --device /dev/fuse --security-opt apparmor=unconfined "$IMAGE_NAME" sleep 600` 启动测试容器
- 验证 /dev/fuse 设备可用：`docker exec "$CONTAINER_NAME" test -c /dev/fuse`，PASS/FAIL
- 验证 user_allow_other 配置：`docker exec "$CONTAINER_NAME" grep -q "^user_allow_other" /etc/fuse.conf`，PASS/FAIL
- 在容器内执行**真实 sshfs FUSE 挂载**验证（模拟 mount.go 中的 sshfs passive 模式）：
  ```bash
  docker exec "$CONTAINER_NAME" bash -c '
    mkdir -p /tmp/fuse-src /tmp/fuse-mount
    echo "fuse-test-content" > /tmp/fuse-src/test.txt

    # 启动容器内本地 SFTP server 用于 sshfs 挂载测试
    # 生成临时 host key 供 sshd 使用
    ssh-keygen -A 2>/dev/null || true
    # 使用 sshfs 挂载本地目录（通过 loopback SSH）验证 FUSE 系统调用链
    ssh-keygen -t ed25519 -f /tmp/fuse-testkey -N "" -q
    mkdir -p /root/.ssh
    cat /tmp/fuse-testkey.pub >> /root/.ssh/authorized_keys
    chmod 600 /root/.ssh/authorized_keys

    /usr/sbin/sshd -p 2299 -o ListenAddress=127.0.0.1

    sshfs root@127.0.0.1:/tmp/fuse-src /tmp/fuse-mount \
      -p 2299 \
      -o StrictHostKeyChecking=no \
      -o IdentityFile=/tmp/fuse-testkey \
      -o allow_other

    # 使用 mountpoint -q 验证挂载就绪（与 mount.go waitForMount 一致）
    if mountpoint -q /tmp/fuse-mount; then
      echo "SSHFS_MOUNT_OK"
    else
      echo "SSHFS_MOUNT_FAIL"
    fi

    # 读写验证（SC-1：挂载成功且读写正常）
    if [ -f /tmp/fuse-mount/test.txt ] && [ "$(cat /tmp/fuse-mount/test.txt)" = "fuse-test-content" ]; then
      echo "READ_OK"
    else
      echo "READ_FAIL"
    fi

    echo "write-verify" > /tmp/fuse-mount/write-test.txt
    if [ -f /tmp/fuse-src/write-test.txt ] && [ "$(cat /tmp/fuse-src/write-test.txt)" = "write-verify" ]; then
      echo "WRITE_OK"
    else
      echo "WRITE_FAIL"
    fi

    # 清理（与 mount.go fusermountCleanup 对齐）
    fusermount -u /tmp/fuse-mount 2>/dev/null || umount /tmp/fuse-mount 2>/dev/null || true
  '
  ```
  根据输出判断：
  - SSHFS_MOUNT_OK → PASS "sshfs FUSE 挂载: 成功（mountpoint -q 确认）"
  - READ_OK → PASS "FUSE 挂载读取: 成功"
  - WRITE_OK → PASS "FUSE 挂载写入: 成功"
  - 任一 FAIL → FAIL 并提示检查 AppArmor 配置和 FUSE 设备权限

**阶段 3：网络策略共存验证**（per D-06, D-07, SC-2）：

前置检查（per D-07）：检测宿主机是否已启用全隧道出网。检查方式：
  ```bash
  # 检测 nftables 默认拒绝规则是否存在
  if nft list chain inet cloud_cli_proxy forward 2>/dev/null | grep -q "drop"; then
    info "全隧道出网: nftables 默认拒绝规则已启用"
    TUNNEL_ACTIVE=true
  else
    warn "全隧道出网: nftables 默认拒绝规则未检测到（D-07 要求在全隧道状态下验证）"
    TUNNEL_ACTIVE=false
  fi
  ```
- 打印说明：sshfs slave 模式的 SFTP 数据走 SSH session channel（进程内 pipe），不经过容器网络栈
- 检查容器内 sshfs 命令可用：`docker exec "$CONTAINER_NAME" command -v sshfs`
- 检查 nftables 规则状态：`nft list ruleset 2>/dev/null | head -5` 或提示 nft 不可用
- 如果 TUNNEL_ACTIVE=true 且阶段 2 的 sshfs 挂载成功 → PASS "FUSE 与网络策略: 全隧道状态下共存正常（per D-07）"
- 如果 TUNNEL_ACTIVE=false 且阶段 2 的 sshfs 挂载成功 → WARN "FUSE 挂载成功但全隧道出网未启用，建议在生产环境重新验证（per D-07）"
- 如果 sshfs 挂载失败 → FAIL "FUSE 与网络策略: 挂载失败，无法验证共存性"

**阶段 4：端到端流程验证**（per D-02 SC-3）：

  - 打印手工 E2E 验证步骤：1) 启动控制面 2) `cloud-claude connect <host>` 3) 容器内 `mountpoint -q /workspace` 4) 读写文件 5) 确认 Claude Code 可正常启动
