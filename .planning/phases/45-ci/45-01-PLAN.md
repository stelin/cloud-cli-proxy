<rationale>
<!-- 关键设计抉择，回答 executor 可能产生的疑问 -->

1. **为什么用 `//go:build e2e` 构建标签？**
   - 现有 ci.yml go-test job 跑 `go test ./...`；若直接把 e2e 包混进去，会把 testcontainers + docker 依赖拖进每条 PR；e2e 又要求 docker daemon 可用，在 hosted runner 上拉镜像 + 起容器至少多 30s。
   - 加 build tag 后，默认 `go test ./...` 跳过 e2e 包，行为零回退；Plan 05 的新 e2e.yml 显式用 `-tags=e2e` 才触发。

2. **为什么 BaseSuite 放在 harness 子包而不是 tests/e2e/ 根？**
   - tests/e2e/ 根放 *_test.go（带 build tag），按 Go 测试包惯例只允许 `package e2e` 或 `package e2e_test`。
   - 后续 Plan 02/03/04 会往 harness 加 Scenario、waitFor、artifact dump 等可复用代码，独立成包便于其它子目录（如 tests/e2e/leak/）import。
   - 包名 = `harness`，文件 `tests/e2e/harness/suite.go` 不带 `_test.go`，因此带 build tag 后仍能被任意 `package e2e_xxx` import。

3. **为什么烟雾用例只起 postgres，不起控制面？**
   - 控制面启动需要 migration、admin JWT secret、SSE hub 一整套；属 Plan 02 Scenario 抽象的职责。
   - 本 plan 的目标是「证明 testcontainers + testify/suite 链路能跑」，最小代价验证依赖落地正确。

4. **为什么是 postgres:18 镜像？**
   - 与 PROJECT.md 技术栈一致（PostgreSQL 18.x）；与 scripts/uat-bypass-fixture-up.sh 默认 PG_IMAGE 一致；可直接复用现有 fixture 镜像缓存。
</rationale>

       ```go
       // Package harness 提供 Cloud CLI Proxy e2e 套件的可复用基础设施。
       // 当前包含：BaseSuite（生命周期 hook）。
       // 后续 plan 会陆续补充 Scenario builder（Plan 02）、waitFor helper（Plan 03）、
       // artifact dump（Plan 04）。
       package harness

       import (
           "context"
           "log/slog"
           "os"
           "path/filepath"
           "runtime"
           "testing"

           "github.com/stretchr/testify/suite"
       )

       // BaseSuite 是所有 e2e suite 的基础。它提供：
       //   - Ctx / Cancel：suite 生命周期上下文，SetupSuite 创建、TearDownSuite 取消。
       //   - Logger：结构化日志器，key-value 风格与项目主代码保持一致。
       //   - ProjectRoot：仓库根目录绝对路径（基于 runtime.Caller 解析），便于
       //     fixture 文件用项目相对路径定位（禁止硬编码 /Users/.../...）。
       //
       // 子 suite 通过结构体嵌入 *BaseSuite 复用以上字段，
       // 并按需在自己的 SetupSuite/TearDownSuite 中先调用基类版本：
       //
       //   func (s *MySuite) SetupSuite()    { s.BaseSuite.SetupSuite() }
       //   func (s *MySuite) TearDownSuite() { s.BaseSuite.TearDownSuite() }
       type BaseSuite struct {
           suite.Suite

           Ctx         context.Context
           Cancel      context.CancelFunc
           Logger      *slog.Logger
           ProjectRoot string
       }

       // SetupSuite 在整个 suite 跑第一个用例之前执行一次。
       func (s *BaseSuite) SetupSuite() {
           s.Ctx, s.Cancel = context.WithCancel(context.Background())
           s.Logger = slog.New(slog.NewTextHandler(os.Stderr, &slog.HandlerOptions{Level: slog.LevelInfo}))
           s.ProjectRoot = projectRootFromCaller()
           s.Logger.Info("e2e BaseSuite ready", "project_root", s.ProjectRoot)
       }

       // TearDownSuite 在整个 suite 跑完最后一个用例之后执行一次。
       func (s *BaseSuite) TearDownSuite() {
           if s.Cancel != nil {
               s.Cancel()
           }
       }

       // SetupTest 留作 Plan 02+ 的 Scenario 注入点（当前为空）。
       func (s *BaseSuite) SetupTest() {}

       // TearDownTest 留作 Plan 04 失败 artifact dump 钩点（当前为空）。
       // Plan 04 会在此检查 s.T().Failed() 并调用 dump.Collect(...)。
       func (s *BaseSuite) TearDownTest() {}

       // Require 是 require.Assertions 的快捷访问，行为与 testify/suite 一致；
       // 这里显式声明便于 IDE 跳转与文档生成（实际由 suite.Suite 提供）。
       var _ = func(_ *testing.T) {} // 保留 testing 包导入用于未来扩展

       // projectRootFromCaller 通过 runtime.Caller 反推仓库根目录（即 go.mod 所在目录）。
       // 不依赖 git，也不依赖 CWD（go test 在不同目录下 CWD 不稳定）。
       func projectRootFromCaller() string {
           _, file, _, _ := runtime.Caller(0) // 当前文件：tests/e2e/harness/suite.go
           // 向上 3 级回到仓库根：harness/ → e2e/ → tests/ → <root>
           return filepath.Clean(filepath.Join(filepath.Dir(file), "..", "..", ".."))
       }
       ```

       ```go
           req := testcontainers.ContainerRequest{
       ```
