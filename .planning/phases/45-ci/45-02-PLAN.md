type Scenario struct {
    t            *testing.T
    logger       *slog.Logger
    projectRoot  string

    // 声明阶段累积的目标拓扑
    controlPlane *controlPlaneSpec
    gateways     map[string]*gatewaySpec
    hosts        map[string]*hostSpec
    users        map[string]*userSpec

    // Start 后填充的运行时句柄
    pgContainer   testcontainers.Container
    cpHandle      *ControlPlaneHandle
    gatewayHandles map[string]*GatewayHandle
    hostHandles    map[string]*HostHandle
    userHandles    map[string]*UserHandle

    started bool
    cleanups []func(context.Context) error // LIFO
}

type ControlPlaneHandle struct {
    Addr       string  // http://127.0.0.1:<port>
    AdminToken string
    DBURL      string  // postgres://...
}

type GatewayHandle struct {
    Name         string
    HostID       string
    ContainerID  string
    GatewayIP    string  // 10.99.<x>.2
    ConfigDir    string  // network.GatewayConfigDir(HostID)
}

type HostHandle struct {
    ID            string  // DB row id
    Name          string  // logical name (alpha/beta/…)
    ContainerName string  // cloudproxy-<host_id>
}

type UserHandle struct {
    ID            string
    Username      string
    EntryPassword string  // 仅 e2e 用，明文
}

Start 流程（每步失败 → cleanups LIFO 跑一遍 → 返回错）：
  1. 起 postgres testcontainer（postgres:18），WaitFor "ready to accept connections"
  2. 执行 fixtures/postgres-seed.sql（如有非空内容）
  3. 起控制面：推荐 in-process（直接 import internal/controlplane/app 然后跑 app.Start(ctx) 监听随机端口），
     避免重新 build 二进制；fallback 选项是 docker run 控制面镜像（但需要 v3.6 之前 ship 控制面镜像；本 plan 先走 in-process）
  4. 通过控制面 admin API 创建 user/egress IP/host 三件套（可参考 scripts/uat-bypass-fixture-up.sh 调用顺序）
  5. 对每个声明过的 gateway，调用 network.NewContainerProxyProvider(logger).PrepareGateway(ctx, spec) → 等就绪
  6. 对每个声明过的 host，调用 provider.PrepareHost(ctx, spec) → 等 verify 通过
  7. started = true，返回 nil

Stop 流程（LIFO 跑 cleanups）：
  - cleanups 在 Start 内每完成一步就 append 一个对应的回滚 func
  - 失败时 best-effort 调用所有 cleanups，logger.Warn 不阻断
