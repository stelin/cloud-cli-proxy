# Phase 47 Plan 01: 到期容器自动停止 + host.stopped 审计事件 (MVS-06)

## Goal

把「用户 `expires_at` 过期 → ExpiryScanner 触发 → 用户容器被 stop → `events` 表落审计行」这条治理路径变成 e2e 自动用例，作为 MVS-06 治理不变量的真实证据。失败一次就阻塞合并；成功一次就证明 ExpiryScanner + runtime QueueHostAction + events 表写入在串起来仍能跑通。

不在本 plan 范围：

- 出口 IP 双绑互斥（Plan 02）
- host-agent 心跳与恢复（Plan 03）
- 容器优雅停止 vs 强杀语义（CONTEXT.md §Deferred Ideas，本 plan 只断言「停了」+「事件入库」）

## Files to touch

新增 / 修改（除 `helpers_test.go` 与 `helpers.go` 新增片段外，均带 `//go:build e2e && linux`）：

- `tests/e2e/helpers.go`（无 build tag）：新增 `ExpiryEventType` 锁定常量、`SimulateExpiryParams` 纯结构、`ParseEventTypes` 纯函数（用于扫 events 列表）。
- `tests/e2e/helpers_test.go`（无 build tag）：新增 `TestHelpersExpiryEventType_Locked` 与 `TestHelpersParseEventTypes_*` 系列断言。
- `tests/e2e/helpers_linux.go`（带 tag）：在 `*GoldenPath` 上挂 `SimulateExpiry(ctx, userID string, waitForTick bool) error` 方法（直接连 Postgres `UPDATE users.expires_at = now - 1s`，可选等一个 `EXPIRY_SCAN_INTERVAL` tick）。
- `tests/e2e/expiry_test.go`（带 tag）：新增 `TestExpiry_AutoStop_GoldenPath` 主用例。
- `tests/e2e/fixtures/expired-user-host.sql`（可选）：种子 SQL，注入一个 `expires_at` 在 30 天后的正常 user + 一个绑定的 running host；用例运行时再调 `SimulateExpiry` 把时间拉到过去。如果 Plan 02 (Phase 46) 已通过 admin API 完成等价 fixture，可直接复用其通路。
- `cmd/control-plane/main.go`（小幅 production 修改）：把 `EXPIRY_SCAN_INTERVAL` 环境变量解析挂到 `app.Config.ExpiryScanInterval`。当前 `main.go` 没有该 env 解析；这是「ROADMAP 偏差」的根本修复点（详见 §Risk & Mitigation）。

可能改动（仅在硬性需要时）：

- `tests/e2e/harness/scenario.go`：当前 Scenario 不支持把任意环境变量传给控制面子进程。本 plan 添加 `WithControlPlaneEnv(envs map[string]string) *Scenario` builder 方法，把 `EXPIRY_SCAN_INTERVAL=1s` 之类的覆盖传下去。Step 2..7 仍是 sentinel，本 plan 只增 builder + 字段，**不动 Step 2..7 实现**。

不动：`go.mod`（不引入新依赖）；不动 ExpiryScanner 业务逻辑（只通过 env var 缩短轮询间隔）。

## Steps

1. **grep 锁定 source-of-truth**
   - `internal/controlplane/scheduler/expiry.go` 中：`expireUser` 触发后写入的事件 `type` 是 `user.expired`（用户级别）与 `host.stop.expired`（host 级别），不是 ROADMAP / CONTEXT 草案里的 `host.stopped`。本 plan 以源码为准：测试断言 `host.stop.expired` 出现，并在 SUMMARY 中明确记录这一 ROADMAP 偏差。
   - `host.stop.expired` 行 metadata 含 `operator=system`、`reason="user expired"`、`host_id`、`user_id`。

2. **纯函数 + 锁定常量**
   - 在 `tests/e2e/helpers.go` 新增：
     ```go
     // ExpiryEventType 是 Phase 47 Plan 01 锁定的「过期触发停止主机」审计事件类型。
     // 源码 internal/controlplane/scheduler/expiry.go expireUser 写入该值。
     // ROADMAP / CONTEXT 草案曾写作 host.stopped；以源码为准。
     const ExpiryEventType = "host.stop.expired"
     const UserExpiredEventType = "user.expired"

     // ParseEventTypes 抽取 admin events API 响应 body 中的 type 列表（顺序保留）。
     // 输入为 GET /v1/admin/events 的 JSON body 切片。
     func ParseEventTypes(body []byte) ([]string, error)
     ```
   - 在 `tests/e2e/helpers_test.go` 新增对应单测：常量值锁定 + ParseEventTypes 覆盖空、单条、多条、坏 JSON 4 个分支。

3. **GoldenPath.SimulateExpiry**
   - 在 `helpers_linux.go` 新增方法：
     ```go
     func (p *GoldenPath) SimulateExpiry(ctx context.Context, userID string, waitForTick bool) error
     ```
   - 实现：通过 `pgx` 直接连 `p.Scenario.ControlPlane().DBURL` 执行 `UPDATE users SET expires_at = NOW() - INTERVAL '1 second' WHERE id = $1`；若 `waitForTick=true`，调 `harness.WaitFor` 轮询 `events` 表里是否出现 `type='user.expired' AND user_id=$1`，等到出现为止（timeout 30s）。
   - 不直接调 `expiryScanner.Scan()`；通过缩短 `EXPIRY_SCAN_INTERVAL=1s` 让真实生产路径触发。

4. **Scenario 扩展 `WithControlPlaneEnv`**
   - 在 `harness/scenario.go` 新增：
     ```go
     func (s *Scenario) WithControlPlaneEnv(envs map[string]string) *Scenario
     ```
   - 字段挂到 `controlPlaneSpec.ExtraEnv map[string]string`；Step 2 真实实现（Phase 46 Plan 01 完成）后会把它注入子进程 env。
   - 当前 Step 2..7 仍 sentinel，本 plan 只增字段 + builder，不动 Step 2..7 实现。

5. **`main.go` 增加 EXPIRY_SCAN_INTERVAL 解析**
   - 在 `cmd/control-plane/main.go` 增加：
     ```go
     if v := os.Getenv("EXPIRY_SCAN_INTERVAL"); v != "" {
         if d, err := time.ParseDuration(v); err == nil {
             cfg.ExpiryScanInterval = d
         } else {
             slog.Warn("invalid EXPIRY_SCAN_INTERVAL, falling back to default", "value", v, "error", err)
         }
     }
     ```
   - 不修改 `app.Config` 或 `app.Run` 的语义（已经有 default 60s 兜底）。

6. **e2e 主用例 `TestExpiry_AutoStop_GoldenPath`**
   - 调 `StartGoldenPath(t)`，要求 scenario 通过 `WithControlPlaneEnv({"EXPIRY_SCAN_INTERVAL":"1s"})` 启动控制面（当前 Step 2..7 sentinel → 用例 t.Skip）。
   - 调 `g.SimulateExpiry(ctx, g.User.ID, true)`。
   - `harness.WaitFor` 30s 内轮询：
     - 容器 `cloudproxy-<host_id>` 通过 admin API `GET /v1/admin/hosts/{X}` 查看 `status` 字段变 `stopped`；
     - `events` 表里出现 `host.stop.expired AND host_id=<host_id>` 一行，metadata 含 `reason="user expired"` 子串。
   - 双断言：两者必须**同时**成立才算 PASS；任一缺失即 FAIL。
   - 总 timeout 60s（30s 给 scanner 触发 + 30s 缓冲容器停止 + 事件写入）。

7. **darwin 兜底**
   - `expiry_test.go` 首行 `//go:build e2e && linux`；darwin 不参与编译。
   - `helpers.go` 新增片段无 build tag，darwin 上 `helpers_test.go` 中新增单测必须 PASS。

## Risk & Mitigation

| 风险 | 缓解 |
|------|------|
| `EXPIRY_SCAN_INTERVAL=1s` 在重负载 CI 上仍可能超过 60s 总 timeout | timeout 已留 30s 缓冲；如果 Linux runner 持续抖动，SUMMARY 中提议把 timeout 提到 90s 而不是动 scanner 间隔。 |
| ROADMAP 草案写 `host.stopped`、实际源码 `host.stop.expired` | 以源码为准，常量锁在 `helpers.go::ExpiryEventType`；任一漂移立即 darwin 单测期失败。SUMMARY 中明确「ROADMAP 偏差」节。 |
| `cmd/control-plane/main.go` 修改属于「生产代码改动」 | 这是纯 config 解析、不改 Scanner 语义，且对生产部署有正向收益（运维想调间隔不用重编译）；在 SUMMARY 中说明这是 Phase 47 唯一允许的 production 改动。 |
| 直接连 Postgres 改 `expires_at` 绕过了 admin API | CONTEXT.md §Area 1 明确要求「DB 预置过期 fixture」，因为没有 admin API 可以把 `expires_at` 设到过去；这是合理的测试 hack，不影响生产正确性证据。 |
| `WithControlPlaneEnv` 在 Step 2..7 仍 sentinel 时无实际效果 | 接受。Step 2..7 由 Phase 46 Plan 01 后续推进；本 plan 只把 API 锁定好，避免 Linux runner 落地时再改 Scenario 签名。 |
