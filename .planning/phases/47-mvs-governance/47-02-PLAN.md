# Phase 47 Plan 02: 出口 IP 双绑互斥 (MVS-07)

## Goal

把「出口 IP X 已绑给 Host A → 再次 POST 把 X 绑给 Host B 必须 4xx + 稳定 error code + 原绑定不破坏」这条治理路径变成 e2e 自动用例，作为 MVS-07 治理不变量的真实证据。

不在本 plan 范围：

- 到期容器自动停止（Plan 01）
- host-agent 心跳与恢复（Plan 03）
- 并发争抢同一 IP（CONTEXT §Deferred Ideas，本 plan 单 A/B 串行）

## Files to touch

新增 / 修改（除 `helpers.go` / `helpers_test.go` 外，均带 `//go:build e2e && linux`）：

- `tests/e2e/helpers.go`（无 build tag）：
  - 新增 `BindEgressIPResponse` 结构（status int + error code string + raw body []byte）。
  - 新增 `ParseBindEgressIPResponse(status int, body []byte) (BindEgressIPResponse, error)` 纯函数。
  - 新增 `EgressIPDoubleBindContract` 锁定表：列出 4 种 status×errorCode 组合及当前源码实际返回值。
- `tests/e2e/helpers_test.go`（无 build tag）：新增 `TestHelpersParseBindEgressIPResponse_*` 与 `TestHelpersEgressIPDoubleBindContract_Locked` 单测。
- `tests/e2e/helpers_linux.go`（带 tag）：在 `*GoldenPath` 上挂：
  - `AdminLogin(ctx) (token string, err error)` —— 如果 GoldenPath 还没有现成的 admin token 字段，复用 Phase 46 Plan 02 的 admin login 通路。

- `tests/e2e/egress_ip_binding_test.go`（带 tag）：新增 `TestEgressIPBinding_DoubleBindExcluded` 主用例。

不动：`go.mod`；不动 `internal/controlplane/http/admin_bindings.go`（本 plan 只测，不修源码）；不动 `internal/store/migrations/`。

## Steps

1. **grep 源码取真相**
   - 路由：`POST /v1/admin/bindings`，请求体 `{"host_id":"...","egress_ip_id":"..."}`（参见 `internal/controlplane/http/admin_bindings.go` `Bind()` 实现）。
   - 当前返回码（以源码为准）：
     - `400`：缺 host_id / egress_ip_id；body invalid。
     - `404`：host 不存在。
     - `409`：host status='running'，必须先停。
     - `500`：`BindEgressIPToHost` 任何 DB 错（包含已存在绑定时 UNIQUE 冲突，因为只对 `(host_id, egress_ip_id)` UNIQUE，不是对 `egress_ip_id` 单列 UNIQUE）。
   - **关键发现（ROADMAP 偏差）**：`host_egress_bindings` 表只有 `UNIQUE (host_id, egress_ip_id)` 复合约束（见 `0001_initial.sql:39`），没有「同一 egress_ip_id 只能绑给一个 host」的硬约束。所以 MVS-07「IP 已绑 A，再绑 B」可能：
     - 第一种可能：DB 直接成功（行级允许） → 测试断言失败 → 必须修源码补 UNIQUE 或预检查。
     - 第二种可能：handler 因业务校验拒绝（grep 发现暂无此校验）。
   - 本 plan 的测试**写期望行为**（4xx + 稳定 error code），观察当前源码是否能满足；不能满足则把测试标 t.Skip + SUMMARY 标记 backend 缺口。

2. **纯函数 + 锁定表**
   - 在 `tests/e2e/helpers.go`：
     ```go
     // BindEgressIPResponse 是 POST /v1/admin/bindings 解析后的契约视图。
     // ErrorCode：当前源码用 {"error":"..."} 文本（无稳定枚举），先以 raw message 子串落地，
     // 后续若 backend 引入稳定 error code（如 egress_ip_already_bound），更新本结构 + 锁定表。
     type BindEgressIPResponse struct {
         Status       int
         ErrorMessage string
         RawBody      []byte
     }

     func ParseBindEgressIPResponse(status int, body []byte) (BindEgressIPResponse, error)

     // EgressIPDoubleBindContract 锁定「期望」的 4xx + error code 契约。
     // Status: 期望 409，因为这是 conflict 语义（参考 GET/PUT/DELETE 现有 conflict 路径）。
     // 当前源码若返回 500，SUMMARY 列为 backend 缺口，需 Phase 51 QUAL-04 修。
     var EgressIPDoubleBindContract = struct {
         WantStatus       int
         WantErrSubstring string
     }{
         WantStatus:       409,
         WantErrSubstring: "already bound",
     }
     ```
   - 在 `helpers_test.go`：4 个分支单测（标准 JSON / 空 body / 非 JSON / status only）。

3. **GoldenPath 扩展**
   - `AdminLogin`：从 `helpers_linux.go` 既有 admin 通路提取（若 Phase 46 Plan 02 已落，复用；没有则在本 plan 新建）。
   - `PostBindEgressIP`：组装 JSON body、加 Bearer header、解析 response。
   - `QueryBindingExists`：通过 `GET /v1/admin/hosts/{hostID}`（admin_hosts.go::Get() 返回 HostDetail，包含 egress IP 绑定列表）；如该 endpoint 不便取，用 pgx 直查 `SELECT 1 FROM host_egress_bindings WHERE host_id=$1 AND egress_ip_id=$2`。

4. **e2e 主用例 `TestEgressIPBinding_DoubleBindExcluded`**
   - `StartGoldenPath(t)` 获 host A（默认拓扑）。
   - 通过 admin API 再创建 host B（同一 user，或新创建一个 user + host；与 A 不同 host_id）；同样通过 admin API 创建 egress_ip X，绑给 A，确认成功。
   - 关键步骤：`status, errMsg, err := g.PostBindEgressIP(ctx, hostB.ID, x.ID)`。
   - 断言 1：`status == 409`（或至少 `status >= 400 && status < 500`）。
   - 断言 2：`errMsg` 含 `"already bound"` 或 `"egress"` 子串（容忍中英文表述差异）。
   - 断言 3：原 A 绑定仍存在 —— `exists, _ := g.QueryBindingExists(ctx, hostA.ID, x.ID); require.True(t, exists)`。
   - 总 timeout 30s。

5. **darwin 兜底**
   - `egress_ip_binding_test.go` 首行 `//go:build e2e && linux`。
   - `helpers.go` / `helpers_test.go` 新增片段无 tag，darwin 单测必须 PASS。

## Risk & Mitigation

| 风险 | 缓解 |
|------|------|
| `host_egress_bindings` 表无 `egress_ip_id` 单列 UNIQUE，无应用层 pre-check | 本 plan 不修源码（避免改动面过大），测试**写期望行为**。观察实际响应：若 500，SUMMARY 标 backend 缺口；若意外通过（DB 允许 + 没人投诉），同样 SUMMARY 标记，要求 Phase 51 决策。 |
| ROADMAP 草案写「稳定 error code」、当前源码只有自由文本 `{"error":"..."}` | 以源码为准。`BindEgressIPResponse.ErrorMessage` 取 message 子串而非枚举码；SUMMARY 中提议 Phase 51 引入稳定 code（如 `egress_ip_already_bound`）。 |
| host A status='running' 时 bind handler 返回 409 conflict | 本 plan 用 stopped host 做 fixture（创建后不 start），跳过 running 拦截路径，专注「IP 已被占」语义。 |
| 通过 admin API 再创建 user/host 需要额外 fixture 通路 | 复用 Phase 46 Plan 02/05 已落地的 admin API 通路；若不可用，本 plan 用 pgx 直接 INSERT 一行 host_b（合法 hack，因为本 plan 只测 binding handler，不测 host 创建流程）。 |
