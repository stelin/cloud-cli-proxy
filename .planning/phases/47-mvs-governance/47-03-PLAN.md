# Phase 47 Plan 03: host-agent 心跳与恢复 (MVS-08)

## Goal

把「`pkill -9 -f host-agent` → 控制面健康检查 30s 内转 unhealthy → 容器内 supervisor 拉起 host-agent → 60s 内自动转回 healthy，无人工干预」这条治理路径变成 e2e 自动用例，作为 MVS-08 治理不变量的真实证据。

不在本 plan 范围：

- 到期容器自动停止（Plan 01）
- 出口 IP 双绑互斥（Plan 02）
- 心跳恢复时间窗的实测分布（CONTEXT §Deferred Ideas，本 plan 锁 60s 上限）
- 主动调用 force resync API（CONTEXT §Area 3：契约是「自动恢复无人工干预」）

## Files to touch

新增 / 修改（除 `helpers.go` / `helpers_test.go` 外，均带 `//go:build e2e && linux`）：

- `tests/e2e/helpers.go`（无 build tag）：
  - 新增 `HostHealthStatus` 枚举（`Healthy / Unhealthy / Degraded / Unknown`）+ `String()`。
  - 新增 `ParseControlPlaneHealth(body []byte) (overall HostHealthStatus, agent HostHealthStatus, err error)` 纯函数：解析 `/healthz` 响应 `{"status":"ok|warning|degraded","checks":{"agent":"ok|unreachable","database":"..."}}`。
  - 新增 `HostHealthRecoveryContract` 锁定表：unhealthy 上限 30s、recovery 上限 60s。
- `tests/e2e/helpers_test.go`（无 build tag）：新增 `TestHelpersHostHealthStatus_String`、`TestHelpersParseControlPlaneHealth_*` 系列。
- `tests/e2e/helpers_linux.go`（带 tag）：在 `*GoldenPath` 上挂：

不动：`go.mod`；不动 control-plane / host-agent 源码（本 plan 不引入「per-host health API」，而是复用现有全局 `/healthz`）。

## Steps

1. **grep 源码取真相 / ROADMAP 偏差**
   - **关键发现**：当前控制面 **没有** `GET /v1/admin/hosts/{X}/health` 端点（grep `internal/controlplane/http/router.go` 与 `admin_hosts.go`，只有 `/v1/admin/hosts/{hostID}` 主资源 + bypass / claude / VNC 子资源；无 health 子路径）。
   - 实际可用：`GET /healthz`（router.go:103，无 admin guard）。响应：
     ```json
     {"status":"ok|warning|degraded","checks":{"database":"ok|<err>","agent":"ok|unreachable"}}
     ```
     当 host-agent 不可达：`checks.agent="unreachable"`，`status="warning"`（degraded 留给 database 故障）。
   - 这是单宿主机部署的事实约束（v1 仅一台 host-agent），用全局 `/healthz` 作为 MVS-08 的真实 source-of-truth 是合理的；多宿主机场景属未来 phase。
   - **ROADMAP 偏差**：以源码为准，本 plan 不调 `GET /v1/admin/hosts/{X}/health`（不存在），改用 `GET /healthz` 的 `checks.agent` 字段。SUMMARY 与 VERIFICATION 显式记录这一偏差。

2. **纯函数 + 锁定表**
   - 在 `tests/e2e/helpers.go`：
     ```go
     type HostHealthStatus int
     const (
         HostHealthUnknown HostHealthStatus = iota
         HostHealthHealthy
         HostHealthUnhealthy
         HostHealthDegraded
     )
     func (s HostHealthStatus) String() string

     // ParseControlPlaneHealth 解析 GET /healthz 响应。
     // 映射：
     //   checks.agent == "ok"         → HostHealthHealthy
     //   checks.agent == "unreachable" → HostHealthUnhealthy
     //   其他 / 缺失                   → HostHealthUnknown
     // 全局 status 同步映射 ok→Healthy / warning→Unhealthy / degraded→Degraded。
     func ParseControlPlaneHealth(body []byte) (overall HostHealthStatus, agent HostHealthStatus, err error)

     // HostHealthRecoveryContract 锁定 MVS-08 时间窗。
     // 任一阈值漂移 → 修源码或调本表 → SUMMARY 解释。
     var HostHealthRecoveryContract = struct {
         UnhealthyWithin time.Duration
         HealthyWithin   time.Duration
     }{
         UnhealthyWithin: 30 * time.Second,
         HealthyWithin:   60 * time.Second,
     }
     ```
   - 在 `helpers_test.go`：覆盖 5 个分支（ok / warning+agent unreachable / degraded+db down / 缺 checks 字段 / 坏 JSON）+ String() 表 + 时间窗常量锁定。

3. **GoldenPath 扩展**
   - `KillHostAgent(ctx)` 实现思路：
     - 拿到 host-agent 所在容器句柄（embedded 模式下 host-agent 与 control-plane 同进程，**KillHostAgent 等价于杀控制面**，不可测 → 本 plan 仅在「非 embedded 模式」下生效，embedded 模式 t.Skip）。
     - 非 embedded：`docker exec <host-agent-container> sh -c 'pkill -9 -f host-agent'`。
     - 不 `docker kill` 整容器，让容器内 supervisor（dumb-init / systemd）拉起 host-agent。
   - `WaitHostHealthStatus(ctx, expected, timeout)`：
     ```go
     return harness.WaitFor(ctx, fmt.Sprintf("agent=%s", expected),
         func(ctx context.Context) error {
             req, _ := http.NewRequestWithContext(ctx, "GET", p.ControlPlaneURL+"/healthz", nil)
             resp, err := http.DefaultClient.Do(req)
             if err != nil { return err }
             defer resp.Body.Close()
             body, _ := io.ReadAll(resp.Body)
             _, agent, perr := ParseControlPlaneHealth(body)
             if perr != nil { return perr }
             if agent != expected { return fmt.Errorf("agent=%s want=%s", agent, expected) }
             return nil
         },
         harness.WithTimeout(timeout),
         harness.WithPollInterval(500*time.Millisecond),
     )
     ```

4. **e2e 主用例 `TestHostAgent_KillRecover_GoldenPath`**
   - `StartGoldenPath(t)`；若 embedded 模式（即 control-plane 与 host-agent 同进程）→ t.Skip。
   - 入口断言：`g.WaitHostHealthStatus(ctx, HostHealthHealthy, 30*time.Second)` 必须立刻成立（基线健康）。
   - `g.KillHostAgent(ctx)`。
   - 断言 1：`g.WaitHostHealthStatus(ctx, HostHealthUnhealthy, HostHealthRecoveryContract.UnhealthyWithin)` PASS。
   - 断言 2：`g.WaitHostHealthStatus(ctx, HostHealthHealthy, HostHealthRecoveryContract.HealthyWithin)` PASS。
   - 全程不调 force resync API。
   - 总 timeout 120s（30 + 60 + 30 缓冲）。

5. **supervisor 类型识别（Claude's Discretion）**
   - 用例不挑剔 supervisor 实现（dumb-init / systemd / supervisord）；只要 60s 内 host-agent 进程被拉起 + `/healthz` `checks.agent="ok"` 即视为 MVS-08 满足。
   - 如 60s 内未恢复，按 CONTEXT §Area 3：本 plan 不在 darwin 上强证，列 Linux runner deferred；SUMMARY 记录可能的 supervisor 实现缺失。

6. **darwin 兜底**
   - `host_agent_recovery_test.go` 首行 `//go:build e2e && linux`。
   - `helpers.go` 新增片段无 tag，darwin 单测 PASS。

## Risk & Mitigation

| 风险 | 缓解 |
|------|------|
| 控制面无 `GET /v1/admin/hosts/{X}/health` 端点 | 以源码为准，复用 `/healthz` `checks.agent`，单宿主机 v1 语义足够；SUMMARY 与 VERIFICATION 明确「ROADMAP 偏差」。Phase 50/51 若引入多宿主机，再补 per-host endpoint。 |
| embedded 模式下杀 host-agent = 杀控制面 → 用例无法测 | t.Skip + log；CI 必须显式以 `HOST_AGENT_MODE!=embedded` 跑。SUMMARY 提示 CI runner 配置。 |
| 容器内 supervisor 实现不一致（dumb-init 默认不重启子进程） | 这是 Linux runner 配置层面的问题，不影响 darwin 编译；如 60s 内不恢复，SUMMARY 标 backend gap，需 Phase 50/51 落 supervisord / s6 / systemd 进 host-agent 镜像。 |
| `/healthz` 没有 admin guard，但本 plan 反正只读 | 接受。如未来加 guard，本测试需补 token；目前直接 GET 即可。 |
| `pkill -9 -f host-agent` 误杀同名进程 | 在生产镜像里 host-agent 是唯一同名进程（grep `internal/agent/server.go`、`cmd/host-agent/`），风险低；如有歧义，改用 `pkill -9 -x host-agent`（精确匹配）。 |
| 单源 `/healthz` 抓取受 HTTP keepalive 影响假阳 | 用 `http.Client{Timeout:2s, Transport:&http.Transport{DisableKeepAlives:true}}`（与 `harness/waitfor.go` `WaitForHTTP` 内联客户端语义一致）。 |
