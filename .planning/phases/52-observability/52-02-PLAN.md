# Phase 52 Plan 02: 5 子目录 README + metadata.txt 完整契约 (OBS-02)

## Goal

把 Plan 01 落地的「采集骨架」配上**面向人**的中文 README 与 metadata.txt 完整契约：

- `tests/e2e/harness/artifacts/{logs,network,docker,postgres,system}/README.md` 5 份**模板**，被 `collect-artifacts.sh` 在每次采集时复制到 `$ROOT/<sub>/README.md`。
- `metadata.txt` 完整字段：`timestamp` / `scenario` / `hostname` / `kernel` / `git_sha` / `runner` / `script_version`。
- 5 份 README 包含：
  - 「里面是什么」一句话总结
  - 「采集命令」具体 shell 行
  - 「典型排障场景」3-5 条
  - 「darwin 上为空怎么办」说明（明确告知 OK，不算 bug）

**不在本 plan 范围**：

- 采集脚本本身逻辑（Plan 01 已落地）。
- DumpHook 内部切换 / CI workflow（Plan 03 落地）。

## Steps

1. **写 `tests/e2e/harness/artifacts/logs/README.md`**（中文）

   结构（≤ 60 行）：

   ```markdown
   # logs/ —— 容器日志

   ## 里面是什么
   `docker logs --tail 500 --timestamps <name>` 的输出，每个 e2e 用例涉及的容器各 1 份。
   命名规则：`<container-name>.log`，例如 `cloudproxy-gw-<host_id>.log`、`cloudproxy-<host_id>.log`。

   ## 采集命令（collect-artifacts.sh）
   ```bash
   docker ps -a --format '{{.Names}}' | while read name; do
     docker logs --tail 500 --timestamps "$name" > "logs/${name}.log" 2>&1
   done
   ```

   ## 典型排障场景
   1. sing-box gateway 启不来 → 看 `cloudproxy-gw-*.log` 末尾错误。
   2. host-agent 心跳掉线 → 看 `cloudproxy-<host_id>.log` 是否有 panic。
   3. 控制面拒绝 admin 请求 → 看 control-plane 容器日志的 4xx/5xx 响应行。
   4. worker 内 SSH 进不去 → 看 worker 容器日志的 sshd 启动行。

   ## darwin 上为空怎么办
   darwin 本地通常没起 docker daemon，本目录会只剩 `_skipped.txt` 占位。OK，不是 bug。
   要在 darwin 上真实采集，先确认 Docker Desktop 在跑、用例真在 docker 里起了容器。
   ```

2. **写 `tests/e2e/harness/artifacts/network/README.md`**

   - 里面是什么：`nft list ruleset` / `ip -o link` / `ip -o addr` / `ip route` / `ip netns list` / `ss -tln`。
   - 采集命令：列对应的 5 行 shell。
   - 排障场景：
     1. 出口 IP 不对 → 看 `ip-route.txt` 是否走 tun0 默认路由。
     2. nft drop 规则没生效 → grep `nft-ruleset.txt` 找 `169.254.0.0/16 drop` / `default deny`。
     3. worker netns 漏建 → 看 `ip-netns.txt` 是否有对应 host_id。
     4. 监听端口冲突 → 看 `listen-tcp.txt`。
   - darwin 上：`ip` / `nft` 不存在，目录大半 `_skipped.txt`；netstat 可能在但格式不同。

3. **写 `tests/e2e/harness/artifacts/docker/README.md`**

   - 里面是什么：`ps.txt`（docker ps -a） / `network-ls.txt` / `inspect-<name>.json`（每容器 1 个）。
   - 采集命令。
   - 排障场景：
     1. 容器没起 → 看 `ps.txt` 的 STATUS 列。
     2. 容器跑在错的网络上 → 看 `inspect-<name>.json` 的 `NetworkSettings.Networks`。
     3. 容器 OOM/CrashLoopBackOff → 看 `inspect-<name>.json` 的 `State.OOMKilled / RestartCount`。
   - darwin 上：取决于 Docker Desktop 是否在跑。

4. **写 `tests/e2e/harness/artifacts/postgres/README.md`**

   - 里面是什么：`schema.sql`（pg_dump --schema-only） / `users.tsv` / `host-egress-bindings.tsv` / `events.tsv`。
   - 采集命令（含完整 `psql -c` 语句）。
   - **隐私守护说明**：本目录**永远不**导出 `password_hash` / `entry_password` / `admin_token` 等敏感列；schema dump 用 `--schema-only`。
   - 排障场景：
     1. user.status='expired' → 看 `users.tsv`。
     2. 双绑互斥失败 → 看 `host-egress-bindings.tsv` 是否同一 egress_ip_id 出现 2 次（Phase 51-09 修复后应不会）。
     3. 心跳事件丢失 → 看 `events.tsv` 末尾的 `type=host.heartbeat.*`。
   - darwin 上：`PG_DUMP_URL/DATABASE_URL` 没设 → 目录空；本机能跑 e2e 控制面时会有真实输出。

5. **写 `tests/e2e/harness/artifacts/system/README.md`**

   - 里面是什么：`uname.txt` / `free.txt` / `df.txt` / `dmesg-tail.txt` / `wait-timeout.txt`（Phase 45 ArtifactDumper 已在用）。
   - 采集命令。
   - 排障场景：
     1. 内核版本不兼容 sing-box tun → 看 `uname.txt`。
     2. testcontainer 内存挤爆 → 看 `free.txt`。
     3. host disk 满 → 看 `df.txt`。
     4. iptables/nft 内核模块加载失败 → 看 `dmesg-tail.txt`。
     5. e2e WaitFor 超时备忘 → 看 `wait-timeout.txt`（Phase 45 落地）。
   - darwin 上：`dmesg` 一般会权限拒绝，写 stderr 占位；`free` 缺失走 `vm_stat`。

6. **扩展 `collect_metadata`**（Plan 01 已建函数）

   完整字段：

   ```bash
   collect_metadata() {
     local sha
     sha="$(git -C "$(dirname "$0")" rev-parse --short HEAD 2>/dev/null || echo unknown)"
     {
       echo "timestamp=$(date -u +%Y-%m-%dT%H:%M:%SZ)"
       echo "scenario=$SCENARIO"
       echo "hostname=$(hostname 2>/dev/null || echo unknown)"
       echo "kernel=$(uname -srm 2>/dev/null || echo unknown)"
       echo "git_sha=$sha"
       echo "runner=${GITHUB_JOB:-local}"
       echo "script_version=${SCRIPT_VERSION:-v1}"
     } > "$ROOT/metadata.txt"
   }
   ```

   并在脚本顶部加：

   ```bash
   readonly SCRIPT_VERSION="v1"
   ```

7. **复制 README 模板逻辑**

   在 `collect-artifacts.sh` 顶部加：

   ```bash
   readonly SCRIPT_DIR="$(cd "$(dirname "$0")" && pwd)"
   readonly TEMPLATE_DIR="$SCRIPT_DIR/artifacts"
   ```

   每个 `collect_<sub>()` 函数末尾追加：

   ```bash
   cp "$TEMPLATE_DIR/<sub>/README.md" "$ROOT/<sub>/README.md" 2>/dev/null || true
   ```

   也可以单独提一个 `copy_readmes()` 在 main flow 里调用一次：

   ```bash
   copy_readmes() {
     local sub
     for sub in logs network docker postgres system; do
       if [[ -f "$TEMPLATE_DIR/$sub/README.md" ]]; then
         cp "$TEMPLATE_DIR/$sub/README.md" "$ROOT/$sub/README.md" 2>/dev/null || true
       fi
     done
   }
   ```

   推荐用 `copy_readmes()` 单函数，更容易测试。

## Risk & Mitigation

| 风险 | 缓解 |
|------|------|
| README 模板位置 `tests/e2e/harness/artifacts/` 与 Phase 45 ArtifactDumper 的 `./out/e2e-artifacts/` 命名容易混淆 | 模板路径**仅仅是模板**（被 git 跟踪），运行时输出走 `<output-dir>/<scenario-id>/<sub>/README.md`（不在仓库内）；命名上模板用 `artifacts/`、运行时用 `e2e-artifacts/`，README 顶部要说明 |
| `git rev-parse` 在 detached HEAD / shallow clone 上失败 | `|| echo unknown` 兜底；CI runner 上 `actions/checkout@v4` 默认 fetch-depth=1 也能拿到 short SHA |
| `cp` 在并发 e2e 用例同名 scenario 时竞写 README | 5 份 README 内容固定模板，重复写不会有真冲突；`cp -p` 不必要 |
| Plan 03 后续可能让 metadata.txt 加 `e2e_test_name` 字段 | 本 plan 锁 7 个字段，新增字段在 SCRIPT_VERSION 升到 v2 时统一加并更新 README |
| 5 份 README 在 PR review 时看不到模板，开发者不知怎么改 | 模板路径放在 `tests/e2e/harness/artifacts/<sub>/README.md`，git 跟踪 + PR diff 直接命中 |
