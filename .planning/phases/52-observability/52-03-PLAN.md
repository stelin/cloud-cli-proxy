## Steps

1. **artifacts.go：Collect() 末尾调脚本子进程**

   位置：`Collect()` 在 `absDir` 计算完后、`return absDir, nil` 之前，插一段：

   ```go
   // Phase 52 OBS-03：尽力调 collect-artifacts.sh 子进程把真实采集结果落到同一目录。
   // 失败不影响 Collect 自身成败（best-effort），但会写 logger.Warn 留痕。
   if scriptErr := d.runCollectScript(ctx, dir, sanitizeName(name)); scriptErr != nil {
       d.logger.Warn("collect-artifacts.sh failed (best-effort, ignored)",
           "name", name, "dir", absDir, "err", scriptErr)
   }
   ```

   新增私有函数：

   ```go
   // runCollectScript 调 tests/e2e/harness/collect-artifacts.sh 把真实采集结果
   // 落到 outDir/scenarioID/<sub>/。脚本路径与 artifacts.go 同目录，用 runtime.Caller
   // 反推（避免依赖 CWD）。
   //
   // 行为：
   //   - 脚本不存在 → 返回 nil（向后兼容 Plan 01/02 未落地的情况）
   //   - 脚本退出非 0 → 返回 wrapped error（调用方决定吞 / 透）
   //   - 30s 硬超时（CONTEXT §Area 1 锁定 ≤30s）
   func (d *ArtifactDumper) runCollectScript(ctx context.Context, outDir, scenarioID string) error {
       _, thisFile, _, _ := runtime.Caller(0)
       scriptPath := filepath.Join(filepath.Dir(thisFile), "collect-artifacts.sh")
       if _, err := os.Stat(scriptPath); os.IsNotExist(err) {
           return nil
       }

       cctx, cancel := context.WithTimeout(ctx, 30*time.Second)
       defer cancel()

       cmd := exec.CommandContext(cctx, "bash", scriptPath, outDir, scenarioID)
       cmd.Stdout = os.Stderr  // 走 testing.T 的 stderr，便于 CI log 检索
       cmd.Stderr = os.Stderr

       if err := cmd.Run(); err != nil {
           return fmt.Errorf("collect-artifacts.sh: %w", err)
       }
       return nil
   }
   ```

   导入新加：`os/exec`、`runtime`（如未导入）；`time` 已在用。

   **关键点**：
   - 把 `outDir` 传成 `dir`（即 `<baseDir>/<sanitizedName>/<timestamp>`），`scenarioID` 传成空字符串或 `sanitizeName(name)`。**重要决策**：脚本接口是 `<output-dir> [scenario-id]`，它会在 `<output-dir>/<scenario-id>/` 下建 5 子目录。我们想要的最终路径是 `<baseDir>/<sanitizedName>/<timestamp>/{logs,network,...}`，所以应传 `outDir=<baseDir>/<sanitizedName>` + `scenarioID=<timestamp>` 让脚本在 `<baseDir>/<sanitizedName>/<timestamp>/<sub>/` 下落地，与 Go 侧 Plan 04 的目录树**完全重合**。
   - 实现侧需要把 `Collect()` 内已有的 `dir = filepath.Join(d.baseDir, sanitizeName(name), timestamp)` 拆开，把 `parent = filepath.Join(d.baseDir, sanitizeName(name))`、`scenarioID = timestamp` 传给脚本。脚本会幂等 `mkdir -p`，原 Go 侧已建的 5 子目录不会被覆盖（脚本里 5 个 `cp README.md` 会重写 Phase 45 占位 README，这是预期行为：Plan 02 模板取代 Plan 04 占位）。

4. **e2e.yml workflow 改造**

   现状（Phase 45 Plan 05 落地）：

   ```yaml
   - name: Upload artifacts on failure
     if: failure()
     uses: actions/upload-artifact@v4
     with:
       name: e2e-artifacts-${{ github.run_id }}
       path: ./out/e2e-artifacts/
       ...
   ```

   插入步骤（**在 Upload 之前**）：

   ```yaml
   - name: Collect e2e artifacts on failure (script)
     if: failure()
     env:
       DATABASE_URL: ${{ env.E2E_DB_URL }}
     run: |
       bash tests/e2e/harness/collect-artifacts.sh \
         ./out/e2e-artifacts \
         ci-${{ github.job }}-${{ github.run_attempt }}
   ```

   说明：

   - 这一步与 Go 侧 `ArtifactDumper.Collect` 写出的 `<test-name>/<timestamp>/` 并存；脚本以 `ci-<job>-<attempt>` 为 scenario-id，与 Go 侧不冲突。
   - `DATABASE_URL` 从 e2e 步骤透传过来；testcontainers 起 PG 后 URL 是动态的，hosted runner 上不一定能拿到，所以**允许这一步**采集到空 postgres 占位（Plan 01 `collect_postgres` 内 `[[ -z "$url" ]] && skip`）。
   - `if: failure()` 与下面的 Upload 步骤一致，确保跑序为「测试跑 → 失败 → script 收一遍 → upload」。

   **注意 GitHub Actions YAML 微妙点**：`env.E2E_DB_URL` 引用语法要求该变量在 step 或 job 级提前 set；当前 e2e.yml `Run e2e suite` 步骤的 env 只有 `CLOUD_CLI_PROXY_E2E_ARTIFACT_DIR`，本 plan **不**强求把动态 testcontainer URL 透传出来——`DATABASE_URL` 没设时脚本会优雅跳过 postgres 采集，文档显式记录这是 deferred-to-CI 待改善项。最终落地行省略 env block 或留空即可。

   也升级 PR 评论步骤：在中文 body 中把「Phase 52 OBS-01..03 接入完整收集逻辑后」改为「Phase 52 OBS-01..03 已落地：artifact 内的 5 子目录现已包含真实采集结果（详见 README.md）」。

## Risk & Mitigation

| 风险 | 缓解 |
|------|------|
| `runCollectScript` 30s 超时拖慢 e2e 总耗时 | CONTEXT §Area 1 锁的就是 ≤30s；脚本本身在 darwin 上一般 < 2s（缺工具大量跳过），ubuntu-24.04 CI 上一般 < 15s |
| 同一目录被 Go 侧 Collect + 脚本两次写 → Plan 02 README 覆盖 Plan 45 占位 | 这是**预期** & 决策：Plan 02 README 更详尽，覆盖 Plan 45 占位是收尾正向行为；artifacts_test.go 中 Phase 45 既有断言「README 含 'Phase 52' 字样」仍成立（Plan 02 README 模板里也有 Phase 52 字样） |
| Plan 02 README 模板里没有 "Phase 52" 字样 → Phase 45 测试 `TestArtifactDumper_CollectCreates5Subdirs` 中断言失败 | Plan 02 步骤 1 已约束每份 README 顶部含 `Phase 52 OBS-...` 字样；Plan 03 单测也复测一次 |
| `collect-artifacts.sh` 子进程在 `t.TempDir()` 之外写文件 | 脚本所有路径走 `$OUT` / `$ROOT` 形参化，传入 `dir = t.TempDir() + sanitizedName/timestamp`，所有写入都在 t.TempDir() 子树内 |
| e2e.yml 新加步骤名字与已有 step name 冲突 | 新步骤名 `Collect e2e artifacts on failure (script)`，已有名是 `Upload artifacts on failure`，无冲突 |
| Phase 47-50 既有用例对 ArtifactDumper.Collect 输出 path 字符串做了断言（不太可能但需排查） | grep 验证：`rg "Collect\(ctx" tests/e2e/` + `rg "wait-timeout" tests/e2e/` 全部仅在 harness 内引用，**未**有 phase 47-50 用例硬编码绝对路径，零破坏 |
| `runtime.Caller` 拿不到脚本路径（编译期路径与运行期路径不一致） | runtime.Caller(0) 返回的是源文件路径，go test 在 darwin / linux 上都用源文件路径，脚本与 `artifacts.go` 同目录，路径关系稳定 |
| 30s 超时的 ctx 来自外层 testing.T.Context() 可能已经被 cancel | 用 `context.WithTimeout(ctx, 30*time.Second)` 派生新 ctx，外层 cancel 触发时脚本快速失败 + 走 Warn 分支，不阻塞 testing 退出 |
| ROADMAP §Phase 52 §Details 中 "100MB 以内" 体积限制本 plan 没做 | Plan 01 `pg_dump --schema-only` + `docker logs --tail 500` 已天然控制；超 100MB 的极端 case 留 SUMMARY 显式记录 deferred-to-后续 milestone |
